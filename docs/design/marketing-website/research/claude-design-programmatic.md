# Claude Design: Programmatic Access Research

**Date:** 2026-06-27
**Scope:** Can our implementation pipeline talk to / extract from / drive the design tool automatically, or must we hand-pass a markdown brief each time?

## 2. What is baoyu-design?

**Repository:** https://github.com/JimLiu/baoyu-design  
**Install:** `npx skills add JimLiu/baoyu-design`  
**License:** MIT

`baoyu-design` is an open-source Agent Skill that packages Claude Design's methodology as a portable, locally-executable design brief. It runs inside Claude Code, Cursor, Codex Agent, or any file-capable agent — without requiring a `claude.ai` subscription or any external service call.

It is **not** claude.design/Anthropic software. It is a third-party reimplementation of the same design methodology as a plain-Markdown skill. The README says: "Run Claude Design locally as an Agent Skill — without claude.ai/design."

**What it produces:** self-contained `.dc.html` files (Design Components) plus an `_ds/<slug>/` design-system bundle (tokens CSS, React component bundle, manifest, adherence rules). All output lands in `designs/<project>/` in the local repo.

## 4. Programmatic surface

### baoyu-design (the tool that produced our files)

**No REST/GraphQL API, no MCP server, no HTTP endpoints of any kind.** It is pure file-based: the skill is plain Markdown plus JSX scaffolds, invoked by the host agent. There is no service to call.

The automation surface is:
- **Skill invocation via Claude Code:** `/baoyu-design` or describe a design task — the agent reads `SKILL.md` and drives the design process.
- **CLI for design-system compilation:** `node <skill>/agents/compile-design-system.mjs designs/<project>` — this re-generates `_ds_bundle.js`, `_ds_manifest.json`, `_adherence.oxlintrc.json` from source components + tokens. Fully headless, no model call.
- **CLI for validation:** `node <skill>/agents/check-design-system.mjs designs/<project>` — read-only checker, no model call.
- **Importing a design system into a project:** `agents/import-design-system.mjs` — copies the `_ds/<slug>/` bundle into a consuming project folder.
- **Asset recording:** `agents/record-asset.mjs` — writes an entry into `_d_meta.json`.

So design-system tooling (compile, check, import) is fully scriptable CLI. The design generation step (producing or updating `.dc.html` pages) requires a model in the loop — there is no headless "generate page from brief" endpoint.

### Anthropic's claude.ai/design

**No public API, no MCP server, no CLI, no SDK.** As of June 2026, Anthropic has stated "Over the coming weeks, we'll make it easier to build integrations with Claude Design" but has not shipped a developer API.

The `/design-sync` command is a **Claude Code slash command** (not an HTTP API). It syncs the local repo's design system into a Claude Design session and vice versa, but only works interactively through the Claude Code or claude.ai interfaces.

The handoff bundle format is proprietary, not publicly documented, not DTCG-compatible, and not consumable by non-Claude toolchains. Multiple sources confirm it only works when Claude Code opens it natively.

Summary table:

| Surface | baoyu-design (our tool) | claude.ai/design (Anthropic) |
|---|---|---|
| REST/GraphQL API | None | None |
| MCP server | None | None |
| CLI | Yes — compile/check/import agents (no-model) | None |
| SDK / npm package | No package; install via `npx skills add` | None |
| Webhooks | None | None |
| Headless design generation | No — requires a model | No — interactive only |
| Export format docs | Yes — `.dc.html` + `_ds/` bundle, open/inspectable | Proprietary bundle, undocumented |
| Programmatic design-system recompile | Yes — `compile-design-system.mjs` | N/A |

## 5. The `.dc.html` format

`.dc.html` is the **Design Component** format defined by `baoyu-design`. It is:

- A standard HTML file with a `<x-dc>` custom element wrapping the template
- A `<helmet>` element for head injections (CSS token imports, script tags)
- Template syntax using `{{ expr }}` interpolations, `<sc-if>`, `<sc-for>` control elements
- Data props defined in a `<script data-dc-script data-props="...">` element
- Rendered client-side by `support.js` (the `dc-runtime` — a React-based runtime that parses the `<x-dc>` template and mounts it)
- `x-import` for importing sibling Design Components by relative URL

The runtime (`dc-runtime`) is MIT-licensed open-source code. The `support.js` in our repo IS the dc-runtime, built from `dc-runtime/src/*.ts`. You can parse `.dc.html` files programmatically with the `parseDcText()` function exported by the runtime.

The `_ds/<slug>/` bundle is also fully inspectable:
- `_ds_manifest.json` — machine-readable component list, token list, font list, starting points
- `_ds_bundle.js` — compiled React component bundle, loaded as a UMD global
- `tokens/*.css` — CSS custom property tokens (colors, typography, effects, spacing, fonts)
- `styles.css` — global CSS entry point
- `_adherence.oxlintrc.json` — oxlint rules encoding which props each component accepts

None of these are published as npm packages. They are file artifacts produced by the `baoyu-design` compiler and checked into the project.

## 6. Practical verdict

### The question

Can the implementation pipeline drive the design tool automatically — submit a brief, get components back — without a human hand-passing markdown each time?

### The answer

**Partially.** There are two distinct automation tiers:

**Tier 1 — Fully scriptable today (no model required):**
Design-system maintenance is already programmable. After a human edits tokens or components in the `_ds/` source:
```bash
node <baoyu-design-skill>/agents/compile-design-system.mjs "Agenta landing page pivot"
node <baoyu-design-skill>/agents/check-design-system.mjs "Agenta landing page pivot"
```
These can be wired into CI or a Makefile. The `_ds_manifest.json` (token list, component contract, font list) can be parsed by any pipeline to extract the current design-system state.

**Tier 2 — Requires a model in the loop (no headless API exists):**
Generating or updating `.dc.html` pages from a brief requires invoking a capable LLM (Claude Opus 4.8 recommended) with the `baoyu-design` skill loaded. There is no REST endpoint to call. The only automation path is to drive Claude Code or another agent programmatically (e.g., via the Claude API + tool use) and give it the skill as context.

### Recommended integration path

**Option B — consume the exported `.dc.html` + bundle as files (what we do now)** is correct and is the intended workflow. The files are fully version-controllable, parseable, and portable.

To make brief hand-off semi-automatic rather than fully manual:

1. **Keep `_ds/agenta-brand-.../` as the authoritative design-system source.** The `_ds_manifest.json` is already machine-readable and could drive a token-sync script or a Storybook/style-dictionary pipeline.
2. **If a new page or component update is needed,** invoke Claude Code with the `baoyu-design` skill from a script using the Claude API's agent mode. The brief can be a file read from a known path (e.g., `handoff/brief-next-page.md`), eliminating the manual copy-paste step.
3. **Wire design-system recompile into CI.** If anyone edits component JSX or token CSS under the `_ds/` source, running `compile-design-system.mjs` + `check-design-system.mjs` can be a pre-commit hook or CI step.

Option (a) — a real API or MCP to call — does not exist today for either baoyu-design or claude.ai/design.

Option (c) — parse the format ourselves — is feasible for reading `.dc.html` structure (`parseDcText()` is available in `support.js`/dc-runtime), but not a path to generating new designs programmatically.

### Summary

No public API or MCP exists for the design tool. The realistic automation path today: drive Claude Code with the baoyu-design skill from a script (brief as a file, output as files), and separately script the design-system compile/check/import CLIs. The file format is open and parseable.
