## 1. Goal and current state

The product pivoted to **agent building**. Onboarding is being reworked around a
**playground-native** flow (the user lands in the playground on an ephemeral agent and
commits it in place) and an **agent-builder** template flow (clicking a template seeds a
builder prompt instead of writing config directly). Arda's PR #5076 shipped the base of
this. This session cleaned up the pre-pivot ("Agenta 1.0") onboarding, rebuilt the EE
post-signup survey, and added implicit intent analytics.

Everything below is **merged into `big-agents`** (tip `810e331443` at handoff time). There
is no open PR from this work. New work should go on fresh GitButler lanes over `big-agents`
(see §9).

## 2. The feature flags (read via `getEnv`, so container env vars work)

Defined in `web/oss/src/components/pages/agent-home/assets/constants.ts`. Both are A/B
toggles kept side by side while the flows are validated.

- **`NEXT_PUBLIC_AGENT_TEMPLATE_BUILDER`** → `TEMPLATE_BUILDER_MODE`. Controls what
  clicking a template does.
  - `true`: skip the config-review drawer, open the playground seeded with the template's
    builder instruction (the agent-builder flow, a prompt, no direct config write).
  - `false`/unset (default): the config-definition setup drawer flow (`openSetup`).
  - Consumed in `agent-home/hooks/useTemplateSelect.ts`.
- **`NEXT_PUBLIC_AGENT_PLAYGROUND_ONBOARDING`** → `PLAYGROUND_NATIVE_ONBOARDING`.
  Controls WHERE onboarding lives.
  - `true`: first-run `/apps` renders `OnboardingEntry`, which redirects a first-run user
    into an ephemeral agent inside `/playground`, committed in place on send.
  - `false`/unset (default, and current dev value): onboarding stays on the agent-home
    page and navigates to the app playground after create.
  - Consumed in `web/oss/src/pages/w/[workspace_id]/p/[project_id]/apps/index.tsx`.

**Removed flag:** `NEXT_PUBLIC_AGENT_ONBOARDING` existed in the PR #5076 description but was
removed at the branch tip. Consequence: with the playground flag off, `/apps` renders
agent-home **unconditionally** (the old `AppManagement` page is gone). This contradicts the
PR #5076 "both flags off = /apps behaves as before" claim; the old home page is deleted.

**Adjacent flags** (agent-chat wiring, not onboarding toggles, already set in dev):
`NEXT_PUBLIC_AGENT_CHAT_SLICE`, `NEXT_PUBLIC_AGENT_CHAT_API`, `NEXT_PUBLIC_AGENT_CHAT_TRACK`.

`getEnv` (`web/oss/src/lib/helpers/dynamicEnv.ts`) reads `window.__env` (generated by
`web/entrypoint.sh`, which does NOT list these new flags) and falls back to `process.env`,
so in the dev container the flags resolve from the container env.

## 6. Implicit intent analytics

Helper: `web/oss/src/components/pages/agent-home/assets/onboardingAnalytics.ts`
(`classifyAgentIntent`, `captureFirstAgentIntent`, `truncateForCapture`).

Event `first_agent_intent`, fired from:
- Template pick (home cards, playground quick-pick, gallery): `source: "template"` + template
  name/mode; `$set first_agent_intent_v1` = template category.
- Composer free-text submit (`useAgentHomeActions.onCreate`, `AgentChatPanel.handleCreateAgent`):
  `source: "composer"` + message (truncated 500 chars); `$set first_agent_intent_v1` =
  `classifyAgentIntent(message)` (buckets: support / research / ops / content / coding / data / other).
- Browse-templates / skip: `source: "browse_templates"` etc., no person-property write.

Verified this session: null-safe when no PostHog key (try/catch + `posthog?.capture?.(...)`),
fires the composer intent even with no template selected, and survives an empty templates
list. Double-fire prevented by instrumenting the specific UI call sites (a template-seeded
builder message fires once as `template`, never as `composer`).

## 9. GitButler workflow (how to land new changes)

Hard rule: everything through GitButler (`but`), never raw `git` for branch/commit/push, no
worktrees, no `git stash`. Lanes over base `big-agents`.

- New lane: `but branch new <name>` (no anchor = parallel stack). Assign files with
  `but rub <cliId> <lane>` (cliIds from `but status`; paths go stale after any mutation).
  Commit with `but commit <lane> --only`, verify with `git show --stat <lane>`, push with
  `but push <lane>` (silent on success — verify `git rev-parse <lane>` == `git ls-remote origin <lane>`).
- Take the BUT-LOCK in `docs/design/agent-workflows/scratch/agent-coordination.md` (§ BUT-LOCK,
  15-min lease) before any `but` mutation; release it after. Commit the lock/board edit to a
  scratch lane (`chore/scratch-sync-*`) before `but pull` (pull needs a clean tracked tree).
- `but pull` rebases lanes on `big-agents` and archives integrated ones. **Gotcha:** after a
  pull, deleted directories re-materialize as **untracked** residue (byte-identical to the
  deleted blobs). `rm -rf` that residue — it's noise, the tip tree is authoritative. This
  session hit it: the `/get-started` page files re-appeared untracked and the route resolved
  again on the dev stack until removed.
- The three lanes from this session (`feat/onboarding-survey-v2`,
  `feat/onboarding-intent-analytics`, `chore/retire-legacy-onboarding`) are merged and archived.
