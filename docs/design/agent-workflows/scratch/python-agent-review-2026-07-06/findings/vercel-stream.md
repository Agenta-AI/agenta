Read alongside: the consuming route (`sdks/python/agenta/sdk/decorators/routing.py:189-235, 346-386`), the event producer (`sdks/python/agenta/sdk/agents/handler.py:181-247`), the runner's event vocabulary (`services/runner/src/protocol.ts:315-362`), the adapter tests (`sdks/python/oss/tests/pytest/unit/agents/test_ui_messages.py`, `adapters/test_vercel_stream_{park,finish_reason,continuation}.py`), and the actual consumer contract: the ai-sdk the frontend ships, `ai@6.0.0-beta.150` (`web/oss/package.json:79`), whose `uiMessageChunkSchema` and `processUIMessageStream` I read in the built package (`web/node_modules/.pnpm/ai@6.0.0-beta.150_zod@4.4.3/node_modules/ai/dist/index.mjs:4145-4305, 4389-4860, 10994-11016, 11240-11360`).

## How it actually works (verified against code)

**The pipeline.** The handler (`handler.py:181-212`, `agent_event_stream`) runs one turn and yields the runner's neutral events as `{"type", "data"}` dicts, appending a corrective terminal `done` when the terminal result's `stop_reason` disagrees with the streamed one (the runner omits `stopReason` on a HITL pause; `handler.py:194-206`). Routing selects the Vercel projection when the request carries `x-ag-messages-format: vercel` (`decorators/routing.py:353-373`), wraps it in `vercel_sse_stream` (SSE framing plus `data: [DONE]`, `sse.py:17-25`), and stamps the protocol headers (`sse.py:10-14`, `routing.py:14-24`). The projection itself is `agent_stream_to_vercel_stream` (`stream.py:254-446`): it emits `start` / `start-step`, translates each event, then `finish-step` / `finish` with the mapped finish reason, usage, and trace id.

**Event coverage.** The adapter handles exactly the runner's `AgentEvent` union (`protocol.ts:315-362`): `message`, `message_start/delta/end`, `thought`, `thought_start/delta/end`, `tool_call`, `tool_result`, `interaction_request`, `data`, `file`, `usage`, `error`, `done`. Coverage is complete; unknown types are dropped silently (forward-compatible, and correct given the client hard-fails on unknown chunk types).

**The client is strict and it hard-fails, not silently.** Every chunk the FE receives is validated against a `z.union` of `z.strictObject`s (`ai/dist/index.mjs:4145-4305`). On a validation failure `DefaultChatTransport` **throws** (`index.mjs:11009-11011`), `Chat.makeRequest` catches, sets `status: "error"`, and calls `onError` (`index.mjs:11328-11341`). So the review brief's premise ("the client rejects them silently") is doc drift against ai@6-beta.150: one malformed frame does not vanish, it **kills the entire turn**, including content already streamed into the bubble. This is load-bearing for severity below: every nullable field the adapter passes into a `z.string()` slot is a whole-turn killer, not a dropped part. The `AgentChatSlice` FE does render the error in-chat (`web/oss/src/components/AgentChatSlice/AgentChatPanel.tsx:84, 361-372`), so client-side failures are at least visible.

**Message identity and HITL continuation.** Each turn's `start` frame carries a unique `messageId`, preferring `msg-{trace_id}` (`stream.py:272-279`); the shared-constant collision this fixed (duplicate React keys, dropped turns) is documented in place. On a HITL resume, the invoke prelude lifts the trailing assistant `UIMessage.id` into `continuation_message_id` (`decorators/routing.py:224-229`) and threads it through (`routing.py:427, 470`), so the resume turn continues the same client message (the ai-sdk `replaceLastMessage` path, `index.mjs:11297-11307`). Verified end to end in code; pinned by `adapters/test_vercel_stream_continuation.py`.

**Tool calls and the HITL park.** A first `tool_call` emits `tool-input-start` + `tool-input-available`; a repeat for a seen id is treated as an arg refresh and re-emits only `tool-input-available` (`stream.py:330-368`), which matches the client state machine (a second `tool-input-start` would reset the part to `input-streaming`, `index.mjs:4594-4626`). A `user_approval` interaction emits the synthesized tool part when none preceded it, refreshes the stable spec name and real args when one did, then emits `tool-approval-request` (`stream.py:449-526`). The name-preference chain (`resolvedName` â†’ spec `name` â†’ ACP `title`/`kind`, `stream.py:594-610`) and the "approval upgrades the name, a later refresh cannot downgrade it" rule (`stream.py:340-346, 474-478`) exist to keep the cross-turn resume key stable; the runner stamps `resolvedName` at `services/runner/src/engines/sandbox_agent/acp-interactions.ts:62-66`. All of this is pinned by `test_vercel_stream_park.py` (717 lines, behavior-level).

**Finish semantics.** `done.stopReason` uses last-non-null-wins so the handler's corrective terminal `done` beats the runner's empty one (`stream.py:421-429`); the raw model stop reason maps onto the closed ai-sdk enum with `paused`/`cancelled` â†’ `other` and a fall-through to `unknown` (`stream.py:22-54`). This matches the schema's `finishReason` enum exactly (`index.mjs:4283-4295`). Usage and trace id ride `finish.messageMetadata` (`stream.py:434-446`).

**Errors.** The runner never emits an `error` event today (`type: "error"` appears only in `protocol.ts:361`); run failures arrive as an `ok:false` terminal record, which `result_from_wire` raises as a sanitized one-line `RuntimeError` (`utils/wire.py:158-168, 54-74`). The projection catches any exception from the event iterator and emits an `error` frame, then stops without `finish` (`stream.py:430-432`); pinned by `test_ui_messages.py:666-677`. The client throws on the `error` chunk (`index.mjs:4842-4844` + the throwing `onError` at `11318-11320`), sets `status: "error"`, and the FE shows it. So a failed run **is** visible to the user; the swallowed-into-a-blank-turn case is specifically the runner-side ok-true-with-empty-output bug (see finding 2).

**Two projections, one live.** `agent_run_to_vercel_parts` (`stream.py:57-251`) is documented as development-only, and that is accurate: no service or API code calls it (verified by grep; only tests and the `ui_message_stream` back-compat alias reference it). The live path is `agent_stream_to_vercel_stream` only.

**The ingest direction.** `messages.py` maps inbound `UIMessage` parts to neutral blocks: text, file, tool parts (typed `tool-{name}`, `dynamic-tool`, and the output-marker types), and both HITL reply shapes â€” the dedicated `tool-approval-response` part and the ai-sdk's inline decision (`state: "approval-responded"` / `output-denied` with `approval.approved`), which becomes the `{approved}` tool-result envelope the runner's `extractApprovalDecisions` keys on (`messages.py:151-196`). `tool-approval-request` parts are dropped on replay (`messages.py:91-92`). Reasoning and `data-*` parts are dropped on ingest (`messages.py:66-104`).

**Doc drift found.** (1) The brief's "silently rejects" is wrong for this ai version; the failure mode is a thrown error and a dead turn. (2) `stream.py`'s `tool-output-denied` branch reads a `denied` field that `protocol.ts`'s `tool_result` does not define and the runner never emits (finding 5). (3) The orchestration prompt lists `adapters/vercel/stream.py` among "untested load-bearing modules" (lane G candidates); that is stale â€” the adapter has ~1,600 lines of behavior tests across four files. What is genuinely missing is schema conformance, not coverage (finding 3).

## Strengths â€” keep this

- **The projection lives in the right layer.** Format negotiation and the agentaâ†’Vercel translation sit in routing, never in the handler; the handler yields neutral events (`stream.py:261-268`, `decorators/routing.py:360-368`). The `vercel/` package is a genuine protocol adapter with the neutral DTOs as the port.
- **The comments carry the why, and they are accurate.** Nearly every non-obvious branch cites the incident that created it (F-040 park-finish, F-024 clobber, the HITL name-drift resume loop, the duplicate-messageId collision) and I verified each against the code on both sides of the wire. This matches the runner review's best-property finding.
- **The HITL sequences are right.** Synthesized tool part before `tool-approval-request` (the client's `getToolInvocation` requires the part to exist, `index.mjs:4403-4416`); no second `tool-input-start` on arg refresh; stable-name preference with an explicit no-downgrade rule; continuation message id so the resume turn folds into the same bubble. This is the hardest part of the adapter and it is correct.
- **Finish-reason discipline.** A closed-set check plus an explicit map plus a safe `unknown` fallback (`stream.py:22-54`) â€” exactly what a strict-enum consumer needs.
- **Sanitized run errors at the wire boundary.** `sanitize_runner_error` keeps one clean line for the UI and logs the full detail (`utils/wire.py:54-74`).
- **Behavior-level tests.** The park/finish-reason/continuation/full-turn-order suites assert frame sequences, not implementation details, and they encode the regression history (F-024, F-040).
- **`AgentStream` fails loud on truncation.** A stream that ends without a terminal record raises instead of leaving `result()` to fail opaquely later (`streaming.py:80-85`).

## Findings

### 1. HIGH â€” Optional runner fields pass straight into required chunk fields; one `null` kills the whole turn client-side

- **Where:** `stream.py:330-368` (tool_call â†’ `toolCallId`/`toolName`), `stream.py:411-416` (file â†’ `url`/`mediaType`), `stream.py:522-526` (`tool-approval-request` â†’ `toolCallId`), `stream.py:528-529, 565-569` (`data-*` â†’ `id`), `stream.py:300-309` (passthrough `text-start`/`delta`/`end` â†’ `id`). Producer optionality: `protocol.ts:326-330` marks `tool_call.id?` and `name?` optional; `acp-interactions.ts:78, 87` can produce a `user_approval` payload with `toolCallId: undefined` and `toolCall: undefined` when the ACP request lacks a tool call.
- **What and why:** the client schema requires `z.string()` for `toolCallId`, `toolName`, `id`, `url`, `mediaType`, and `approvalId`, inside `strictObject`s (`index.mjs:4169-4266`), and `id: z.string().optional()` on `data-*` chunks rejects an explicit `null` (`index.mjs:4264-4271`). The adapter copies `data.get(...)` values through with no normalization, so any absent optional becomes JSON `null` in a required slot. Because validation failure **throws** (`index.mjs:11009-11011`), the cost is not a dropped part: the whole turn errors, including text already rendered.
- **Failure scenario:** the harness announces a tool call whose ACP update carries no id (the protocol allows it) â†’ `tool-input-start` goes out with `"toolCallId": null` â†’ zod rejects â†’ `DefaultChatTransport` throws â†’ the turn flips to `status: "error"` with a cryptic zod message; everything the model already said in that turn is stranded mid-bubble. Same shape for an ACP permission request without a `toolCall` object: `_approval_tool_call_id` returns `None` (`stream.py:613-620`), the approval chunk goes out with `toolCallId: null`, and the park turn dies instead of pausing.
- **Recommendation:** add one normalization gate at the yield boundary (a small `_conform(part)` applied to every yielded dict): synthesize a stable fallback id when a required id is missing (`f"tool-{n}"`, same trick the one-shot text path already uses), coerce `toolName` to a string fallback (`"tool"`), skip a `file` part missing `url`/`mediaType` (log it), and omit `id` keys whose value is `None`. Pair it with finding 3's conformance test so the gate is provably sufficient.
- **Horizon:** short.

### 2. HIGH â€” No egress backstop for the "ok:true, zero content" run: the user gets a blank assistant bubble with a clean `finish` and no error

- **Where:** `stream.py:434-446` (the finish block has no notion of whether any content frame was emitted); the event loop at `stream.py:289-432`.
- **What and why:** a run that streams no `message`/`thought`/`tool_call`/`data`/`file` events but ends `ok:true` produces exactly `start`, `start-step`, `finish-step`, `finish(finishReason: "stop")`. The client pushes the assistant message on the `start` chunk (`messageId != null` triggers `write()`, `index.mjs:4787-4795`), so the user sees an empty bubble that settles as a normal completed turn. This is the Python side of the known "No response" class. **Reconciliation:** the runner review found the runner-side protection for exactly this case (the F-030 swallowed-Pi-error scan) is dead for every run with skills or a system prompt â€” which is always, for the `agenta` harness (engine F6). Confirmed from this side: nothing in the egress compensates. At launch, the primary harness can return ok-true-empty and the playground shows a silent blank turn. The two reviews agree; this is the last line of defense and it is open.
- **Failure scenario:** provider auth failure that Pi swallows (runner F6) â†’ runner returns `ok:true`, no events, `stopReason: "end_turn"` â†’ frames `start`/`start-step`/`finish-step`/`finish(stop)` â†’ empty bubble, `status: "ready"`, no error anywhere. The user retries, burns another run, same result.
- **Recommendation:** track a `content_parts_emitted` counter in the projection. If `finish` would fire with zero content parts and the mapped finish reason is not `other` (a park legitimately has tool/approval parts anyway, but keep the guard conservative), emit `{"type": "error", "errorText": "The agent returned no output."}` before closing (or a `data-notice` part if the team prefers a softer rendering). Ten lines, and it converts every future instance of this bug class from a silent blank into a visible failure. The root fix stays runner-side (F6), but do not ship with both layers open.
- **Horizon:** short.

### 3. HIGH â€” The frames are hand-mirrored against a strict schema in a beta package, with no conformance pinning on either side

- **Where:** every yield in `stream.py`; the consumer schema is `uiMessageChunkSchema` in `ai@6.0.0-beta.150` (`index.mjs:4145-4305`), pinned exactly in `web/oss/package.json:79`.
- **What and why:** the /run wire contract earned golden fixtures asserted by both languages plus a compile-time key guard (the runner review calls it the strongest hand-built guard). The Vercel frames â€” the contract the *user-facing surface* depends on â€” have neither. The adapter tests assert hand-written expected dicts, which pins the adapter against the author's memory of the schema, not against the schema. And the schema is a moving target: the package is a **beta**, `tool-approval-request`/`tool-output-denied` appeared during the 6.0 beta line, and any FE bump of `ai` can tighten or reshape the union. Combined with finding 1's hard-fail semantics, an FE-side upgrade can break every streamed turn with no Python-side signal. This extends runner finding A-9 (the three-mirror contract at its ceiling): this is a fourth hand-kept mirror, against a contract Agenta does not even own.
- **Failure scenario:** the FE upgrades `ai` to a later beta where a chunk field is renamed or a new required field lands â†’ every Python-emitted frame of that type fails validation â†’ every streamed agent turn errors in the playground â†’ nothing in the Python suite went red.
- **Recommendation:** vendor the chunk schema once (export `uiMessageChunkSchema` as JSON Schema from a tiny script in `web/`, or hand-vendor the union as a JSON file with the `ai` version stamped in it) and add one Python test helper `assert_conforms(part)` run over **every** frame the existing suites collect. Wire the vendored file's `ai` version to `web/oss/package.json` in CI (a grep-level check is enough) so a bump forces regeneration. This is the single highest-value test to add for this lane and it also answers lane G's gap.
- **Horizon:** short (the test), medium (the CI version link).

### 4. MEDIUM â€” `tool_result` and skipped-synthesis approvals can reference a tool part the client never saw; the client throws "no tool invocation found"

- **Where:** `stream.py:369-399` (`tool_result` emits `tool-output-available`/`-error`/`-denied` with no `seen_tool_calls` check and no synthesis), `stream.py:495-509` (approval synthesis requires `isinstance(tool_call, dict)`; a payload with a `toolCallId` but no `toolCall` object skips it). Client: `getToolInvocation` throws for an unknown `toolCallId` (`index.mjs:4403-4416`, used by approval/output/denied cases at 4700-4790).
- **What and why:** the adapter already knows the client requires the tool part to exist before any output or approval frame references it â€” that is why approval synthesis exists. The `tool_result` path never got the same protection. On a continuation turn the cloned message usually still holds the part, which hides the gap; a fresh turn where the runner emits a result without its call (an ACP `tool_call_update` arriving without the announce, a replay edge, a future harness) is fatal.
- **Failure scenario:** runner emits `tool_result {id: "call-9", output: "..."}` with no prior `tool_call` for `call-9` in this stream and no matching part in the continued message â†’ `tool-output-available` â†’ client throws `no tool invocation found for tool call call-9` â†’ dead turn.
- **Recommendation:** in the `tool_result` branch, when `tool_call_id not in seen_tool_calls`, synthesize the same `tool-input-start` + `tool-input-available` pair the approval path synthesizes (name from `tool_names_by_id` or a fallback) before the output frame. Do the same for the approval path when `toolCall` is missing but `toolCallId` is present. Note this only helps once finding 1 guarantees the ids are strings.
- **Horizon:** short.

### 5. MEDIUM â€” The `tool-output-denied` branch and its test pin a `denied` wire field that no producer emits

- **Where:** `stream.py:371-375` (and the dev twin at 168-172) read `data.get("denied")`; `protocol.ts:331-339` defines `tool_result` with no `denied` member; grep across `services/runner/src` finds no emission site. The test `test_ui_messages.py:543-560` feeds `denied: True` by hand.
- **What and why:** this is contract drift between the mirrors, the exact failure mode lane B tracks. A real denial today reaches the client as a plain `tool_result` (`isError` or "denied by policy" text, `services/runner/src/tools/relay.ts:322-330`), so the FE renders a denied tool as a generic error/output instead of the ai-sdk's `output-denied` state. The dead branch also misleads: a reader (and the test suite) believes the denied rendering works.
- **Failure scenario:** user denies a gated tool â†’ next turn's stream carries `tool_result {isError: true, output: "Tool 'x' is denied by the permission policy."}` â†’ the FE shows a red error on the tool part, indistinguishable from a tool that crashed, instead of the denied state the UI is built to render.
- **Recommendation:** decide the contract on the runner side: add `denied?: boolean` to `protocol.ts`'s `tool_result` and set it where relay/responder resolve a denial, keeping the Python branch; or delete the branch and its test. Do not leave a consumer for a field with no producer.
- **Horizon:** medium.

### 6. MEDIUM â€” The projection's catch-all emits unsanitized exception text to the UI

- **Where:** `stream.py:430-432` (and 223-225): `yield {"type": "error", "errorText": str(exc)}`.
- **What and why:** the runner-failure path is sanitized at the wire boundary (`utils/wire.py:54-74`), but the catch-all wraps the *whole* event iteration, so transport and adapter exceptions (httpx connect errors with internal runner URLs, file paths from an OSError, a pydantic validation dump) go to the browser verbatim. The runner review's security lane made internal-detail hygiene a theme; this is the one place this lane leaks it. Also `str(exc)` can be empty (`RuntimeError()`), producing an error with no message.
- **Failure scenario:** the runner HTTP transport refuses a connection mid-stream â†’ `errorText: "All connection attempts failed ... http://runner.internal:8790/run"` renders in the playground, exposing internal topology to any workspace member.
- **Recommendation:** route the catch-all through `sanitize_runner_error` (it already handles multi-line and stack-frame noise and falls back to a generic line), and log the full exception with the trace id.
- **Horizon:** short.

### 7. MEDIUM â€” 175 duplicated lines between the live projection and the dev-only twin, and they have already drifted

- **Where:** `stream.py:71-251` (`agent_run_to_vercel_parts`) vs `stream.py:276-446` (`agent_stream_to_vercel_stream`). The bodies are near-identical, comment for comment.
- **What and why:** the twins differ only in event access (attribute vs dict), the terminal-result fallback, and â€” the drift â€” message-id handling: the live path mints a unique `messageId` (`stream.py:272-275`); the dev twin still defaults to the constant `"msg-1"` (`stream.py:61`) that caused the duplicate-turns bug the live path fixed. Every HITL fix so far (seen-id refresh, name no-downgrade, F-040) had to be applied twice, and the comments say so ("mirrors the routing-layer twin"). The next fix will land in one and not the other; the tests mostly exercise the dev twin (`agent_run_to_vercel_parts`), so a live-only regression can pass the suite.
- **Failure scenario:** a future frame fix lands in the dev twin (where the tests point) and not in the live function â†’ tests green, playground broken.
- **Recommendation:** extract one synchronous core, `_project_event(etype, data, state) -> Iterator[part]`, plus a shared prologue/epilogue; keep both entry points as thin wrappers. Point the tests at the live wrapper. This also gives finding 1's normalization gate a single home.
- **Horizon:** medium.

### 8. MEDIUM â€” No SSE keepalive: a long silent gap (a slow tool, a long think) leaves the stream idle and proxy-killable

- **Where:** `sse.py:17-25`; headers at `sse.py:10-14` disable proxy buffering but nothing defends against idle timeouts.
- **What and why:** between `tool-input-available` and the tool's result the stream emits nothing. A multi-minute bash/browser tool or a long provider stall means minutes of silence on an open SSE connection; default idle timeouts in nginx/traefik/ELB commonly sit at 60s. The client sees a network error mid-turn; there is no `abort` frame and no resume (`reconnectToStream` is the transport's no-op default). The runner review's blocker ("no run has a deadline") is the server side of this; the Python egress is where the user-visible symptom lands.
- **Failure scenario:** agent runs a 3-minute build via a tool on Daytona â†’ no frames for 3 minutes â†’ the proxy closes the SSE at 60-90s â†’ the turn dies with a fetch error while the run completes (and bills) server-side.
- **Recommendation:** emit an SSE comment line (`: keepalive\n\n`) every 15s from `vercel_sse_stream` when no chunk has flushed (a `asyncio.wait_for` around the iterator's `__anext__`). Comments are protocol-invisible to `EventSourceParserStream`, so no schema risk.
- **Horizon:** medium (short if launch fronts the playground with a default-timeout proxy).

### 9. LOW â€” `json.dumps` without a fallback can abort the response mid-stream with no error frame

- **Where:** `sse.py:22`, and the twin `_sse_stream`/`_ndjson_stream` in `decorators/routing.py:256-269`.
- **What and why:** every payload on the live path is JSON-derived from the runner today, so this does not fire. But `data`/`tool_result` payloads are passed through verbatim, and a future in-process producer (local tools, the services app) can hand a datetime or bytes into a part; `dumps` raises inside the `StreamingResponse` generator, Starlette drops the connection, and the client sees a raw disconnect after the error-frame machinery has already been bypassed.
- **Recommendation:** `dumps(chunk, ensure_ascii=False, default=str)`.
- **Horizon:** medium.

### 10. LOW â€” Ingest oddities in `messages.py`: a stray uppercase marker literal, reasoning parts silently dropped, and a name-less commit heuristic

- **Where:** `messages.py:26-31` â€” `TOOL_OUTPUT_MARKER_TYPES` contains the literal string `"TOOL_OUTPUT_AVAILABLE"` alongside the real part-type values; no producer emits an uppercase part type, so it is either dead or papering over an unrecorded FE bug. `messages.py:66-104` â€” `reasoning` parts (which the client persists for every thought the egress streamed) fall through to `[]`, so replayed history silently loses all reasoning; fine if intentional, but nothing says so. `stream.py:631-658` â€” `_committed_revision_data` treats `tool_name is None` as "possibly commit_revision", so a result that arrived without a recorded name and happens to carry `{count, workflow_revision}` emits a spurious `data-committed-revision` refresh.
- **Recommendation:** delete the uppercase literal (or comment the incident it guards); add a one-line comment declaring reasoning drop intentional; require `tool_name == "commit_revision"` exactly.
- **Horizon:** medium.

### 11. LOW â€” `AgentStream` niceties: silent cleanup swallow and a misleading re-iteration error

- **Where:** `streaming.py:87-91` â€” cleanup exceptions are swallowed with no log; `streaming.py:64-85` â€” `__aiter__` is an async generator method, so a second `async for` over the same instance builds a fresh generator over the exhausted record source and raises "stream ended without a terminal result record", which misdescribes the caller's mistake (double iteration) as a runner truncation.
- **Recommendation:** log swallowed cleanup errors at warning; set a `_consumed` flag and raise "AgentStream can only be iterated once" on re-entry.
- **Horizon:** long.
