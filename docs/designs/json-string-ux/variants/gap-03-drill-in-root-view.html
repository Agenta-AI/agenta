Gap 03 — Drill-in root view bails to one giant code editor

The editable surface, anchored on the drill-in component itself. The "drawer" in the screenshots is just EntityDualViewEditor wrapping EntityDrillInView wrapping DrillInContent. The fix lives in DrillInContent (and propagates everywhere the drill-in is consumed: testcase drawer, "Add to Testset" preview, playground execution item, annotation UI).

For non-trivial testcases (stringified-nested, deeply nested, messages-with-tools), the drill-in at root renders top-level objects as one big JSON code editor each. Users only get per-property cards after clicking Drill In. The drilled-in experience is good. The default isn't.

Anchor component: DrillInContent ([DrillInContent.tsx:178](../../../web/oss/src/components/DrillInView/DrillInContent.tsx)). Auto-expand top-level keys to first-level cards. Don't touch drawer chrome — the drawer just hosts the drill-in.

Three approaches (proposed)

A. Auto-expand top-level★ recommended

Each top-level key renders as a per-property card. Object/array values render their first level inline. Drill-in unlocks deeper levels.

▾ inputs obj 3 props ⎘</>Drill In ▸
▾ country str
▾ correct_answer str
▾ metadata obj Drill In ▸
{ "source": "trace", "trace_id": "abc123", "latency_ms": 420 }
Tradeoffs
Pros: structure visible immediately. Matches drilled-in UX. Stringified-JSON detection still kicks in (metadata renders as parsed obj).
Cons: very large/deep testcases get a tall scroll. Stops at first level inline; deeper requires drill-in.

B. Threshold-based smart default

Cards if obj has ≤ 3 properties AND depth ≤ 2. Code editor otherwise. User can toggle either way.

▾ inputs obj 3 props · depth 2 → cards </>
[as in A]
▾ outputs obj depth 5 → editor Show as cards
{
  "answer": {
    "primary": "Funafuti",
    "verification": { ... }
  }
}
Tradeoffs
Pros: balances readability with scroll-pain.
Cons: users learn a non-obvious rule. Edge cases surprise ("why does this one show cards but the bigger one doesn't?").

C. Always cards, paginate deep

Always render top-level keys as cards. For object values, show first 5 keys + "show more (5 of 12)".

▾ context obj 8 props
▾ geo obj
▾ demographics obj
▾ languages obj
▸ Show 5 more (8 total)
Tradeoffs
Pros: consistent behavior, scales to large testcases.
Cons: more UI complexity. Pagination chrome adds clutter.
Recommendation: Variant A. Auto-expand top-level objects to first-level cards. Use existing Drill In for deeper nesting. The </> toggle stays available for users who prefer raw JSON. The current default optimizes for the wrong case — users want to see structure first, edit raw second.
