Gap 01 — Type chips (the shared visual vocabulary)

The chip system is the primitive every other gap doc uses. detectDataType drives widget selection inside the drill-in (DrillInContent.tsx:771) — but there's no type chip in the production drill-in today. DrillInFieldHeader (DrillInFieldHeader.tsx:209) renders the field name, an item count ([N properties]), a view-mode <Select> for the storage format (Text / JSON / YAML / Markdown / Raw), copy / drill-in / collapse buttons. The view-mode dropdown is the closest thing to a type indicator today — but it tells the user how the value will be edited, not what shape the value is.

What gap-01 proposes: add a separate [str] / [obj] / [arr] / [null] / [msgs] / [dotted-key] chip on the field header — distinct from the view-mode selector, alongside the count. Pull the same chip vocabulary up into the testset table cell renderer (gap-02) and the variables panel.

Anchor components: DrillInFieldHeader (where the new chip goes — alongside the existing item count + view-mode selector), testset table cell renderer (currently chip-less), variables panel. Verify the live "before" state at /gap-01-type-chips in the design-mockups app — that page mounts the production drill-in unchanged so you can confirm what's actually there.

Two-tier landing strategy: the chip system should land in both places to keep the OSS app working today and the package consumers benefiting going forward.

  • Tier-3 (OSS) — patch web/oss/src/components/DrillInView/DrillInFieldHeader.tsx:209 directly. This is what 6+ OSS surfaces render today (testcase drawer, annotation, observability, "Add to Testset," playground execution item).
  • Tier-2 (package) — provide a FieldHeaderSlot override via DrillInSlots.fieldHeader on MoleculeDrillInView. Slot prop is FieldHeaderSlotProps from @agenta/entity-ui/drill-in. New molecule-first surfaces (evaluator drill-in, future testcase migration) pick up the chip vocabulary automatically.

Today Current state

DrillInFieldHeader renders the field name, an item count (when applicable), a view-mode selector (Text / JSON / YAML / Markdown / Raw), and action buttons (copy / drill-in / collapse / raw-toggle). No type chip. The view-mode dropdown tells the user how the value will be edited, not what shape the value is. Testset table cells (the view-only entry point) have no chip wiring either. Variables panel and span detail also miss it.

The user infers type from widget shape (quoted text input vs code editor) or from glyphs (quotes/braces/brackets). Works for trivial cases. Breaks the moment a value is ambiguous (e.g., a stringified-JSON-as-string vs a real object).

Fixture 01, drill-in Fields view (hosted in the testcase drawer): country, correct_answer, outputs all render as text inputs. No chip indicating they're strings. Type is left to inference.
Fixture 02, drill-in Fields view: inputs and outputs render as code-block widgets. The widget choice IS the type signal — but no explicit chip, no count, no shape preview at the field header.

Three placement strategies (compact reference)

A. Always-on

Every value, every row, every surface. Maximum signal. Mahmoud's RFC literal reading.

Property values
countrystr"Tuvalu"
profileobj{ name: "Ada", … }
agenum42
activebooltrue
tagsarr[ "x", "y", … ]
notesnullnull
Tradeoffs
Pros: consistent. Type info always visible. Literal reading of RFC.
Cons: chip on every string adds visual noise. Strings are most common.

B. Hover-only

Reveal on row hover. Cleaner default state.

Property values hover row 2
country"Tuvalu"
profileobj{ name: "Ada", … }
age42
activetrue
tags[ "x", "y", … ]
Tradeoffs
Pros: minimal visual noise at rest.
Cons: mobile-hostile. Discoverability — users may never hover, never learn the system.

C. Ambiguous-only★ recommended

Strings/numbers/booleans get no chip (rendering self-evident). Objects/arrays/null/messages always show.

Property values
country"Tuvalu"
profileobj{ name: "Ada", … }
age42
activetrue
tagsarr[ "x", "y", … ]
notesnullnull
Tradeoffs
Pros: clean for the common case. Important types stay visible.
Cons: requires "is rendering self-evident?" judgement. RFC's literal reading is "always-on."
Recommendation: Variant C for compact surfaces (table cells, observability/eval display). Always-on for drill-in field headers (DrillInFieldHeader, the editing surface where consistency pays). Compromise that respects both readability and RFC intent.

Note on column headers: the testset table column-grouping / expansion model (see gap-02) creates parent-child column relationships. Chips help disambiguate them — [obj] on a column header signals "this expands"; [dotted-key] on a flat literal column signals "this is a literal-dot key, not a path component" (see gap-05); [mixed] warns when cells have heterogeneous types.
