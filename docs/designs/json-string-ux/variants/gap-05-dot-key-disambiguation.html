Gap 05 — Literal dotted keys vs nested paths

Second correctness gap, sharper edge case. A testcase can store {"geo.region": "..."} (literal flat key) AND {"geo": {"region": "..."}} (nested object). Both are valid. Both are stored faithfully. Templates resolve them differently under literal-key-first.

The column-grouping model already shows them as separate columns. What's missing is labeling — uses the [dotted-key] chip from gap-01 stacked with a collision-warning chip and an inline runtime explanation.

Anchor components: DrillInContent (path navigation already does path.split(".") at line 244 — adjacent to where the chip + collision detector would live), DrillInFieldHeader (chip stack), testset table cell renderer (column header chips). Same primitive, three surfaces.

Today Current state

Structural separation works (column expansion shows both shapes). Labeling doesn't — no chip says "this is a literal-dot key" vs "this is a nested-path component."

Fixture 08 table: literal geo.region is a flat top-level column; nested geo > region is an expanded sub-column. Both visible. But no chip tells the user which one {{geo.region}} resolves to.
Vanuatu drill-in Fields view (the collision): literal geo.region: "LITERAL_DOT_VALUE" as a flat property AND geo object showing region: "NESTED_PATH_VALUE" side-by-side. Both visible. No marker that one shadows the other for templates.
Kiribati drill-in Fields view: source has only nested geo, but the view shows empty geo.region + geo.subregion inputs (column-union projection from gap-04 bleeds into this gap too).

{{geo.region}} on Vanuatu resolves to "LITERAL_DOT_VALUE" (literal-key-first). The user can see both columns but has no way to know which one wins.

The collision case at real proportions. Vanuatu has BOTH "geo.region" (literal flat key) AND geo.region (nested-object path). Both render in the drill-in. Today nothing flags which one wins for templates.

Three disambiguation approaches (proposed)

A. [dotted-key] chip everywhere★ recommended

Every literal-dot key gets a chip distinguishing it from a nested-path component.

Drill-in Fields view:

▾ "geo.region" dotted-keystr
▾ geo obj
{ "region": "NESTED_PATH_VALUE", "subregion": "Melanesia" }

Variables panel:

{{geo.region}}dottedstr"LITERAL_DOT_VALUE"
{{$.geo.region}}pathstr"NESTED_PATH_VALUE"
Tradeoffs
Pros: explicit, doesn't change behavior, user makes informed choice. Uses the same chip system everywhere.
Cons: adds visual chrome to every dotted column.

B. Auto-warn on collision only

Quiet by default. Warning chip appears only when both shapes exist for the same path.

▾ geo.region ⚠ shadows nestedstr
Templates {{geo.region}} resolve this literal value, shadowing geo.region = "NESTED_PATH_VALUE".
▾ geo obj
{ "region": "NESTED_PATH_VALUE", ... }
Tradeoffs
Pros: quiet by default, only fires when there's actual ambiguity.
Cons: users learn the distinction only when they hit a collision. Variables panel still doesn't disambiguate the two insertion options.

C. Display-time merge ✗ rejected

Auto-collapse: pick one shape (always literal, or always nested) for display. Hide the other.

▾ geo obj
{ "region": "NESTED_PATH_VALUE", "subregion": "Melanesia" }
// "geo.region": "LITERAL_DOT_VALUE" hidden from view
Why rejected
Lossy. Lies to the user about what's stored. Breaks templates that depend on the hidden form. Storage shape must remain visible.
Recommendation: Variant A as the baseline (chip everywhere). Variant B layered on top for collisions specifically (warning chip + inline explanation). Both ship together.
