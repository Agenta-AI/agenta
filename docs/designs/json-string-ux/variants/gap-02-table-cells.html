Gap 02 — Testset table cells render objects/arrays poorly

The view-only entry point. Testset table cells are not a drill-in — they're the surface the user scans before clicking through to DrillInContent. Today the cell renderer doesn't apply the chip vocabulary from gap-01: arrays render as raw multi-line JSON eating row height (fixture 03). Mixed-type cells fall back to — em-dash (fixture 05). Deep objects show truncated raw JSON without shape preview (fixture 06).

The detection logic exists (detectDataType) and the drill-in already uses it. The table cell renderer just doesn't. Pulling the same chips up to the cell makes the table-to-drill-in transition visually continuous.

Anchor component: testset table cell renderer (the only true view-only surface that's not a drill-in). Compose with the existing column-expansion model — chips on collapsed cells, sub-columns when the user expands.

Today Current state

The table already has a strong feature: nested object columns are expandable into sub-column groups. context → demographics + geo → coordinates + region + subregion. Drilling down to leaves works well.

The unsolved cases are where the value is NOT a homogeneous nested object: arrays of records, mixed types per column, leaf objects that don't get expanded.

Fixture 06 collapsed: context and answer render as raw JSON in their cells until the user clicks >. Browseable, but not scannable at a glance. The expansion affordance is there but type/shape signal isn't.
Fixture 06 expanded: context > geo drilled down to coordinates, region, subregion as proper columns. This is the working pattern. Leaf scalars render cleanly; nested groups are reachable.
Fixture 05 — the actual gap: same column has strings in some rows and objects/arrays/null in others. Object/array/null rows render —. Expansion can't help here — the column isn't a homogeneous structure to expand into.

So what's the gap? Two specific cases the existing column-expansion model doesn't solve:

  1. Heterogeneous-type columns (fixture 05). Expansion needs a homogeneous shape; mixed types fall back to —. Need a per-cell type signal + drill-in.
  2. Arrays (fixture 03 languages, neighbors). Expansion would need to invent a row-per-element table-in-cell, which doesn't fit the grid model.
  3. Collapsed leaf-object cells (fixture 06 context before clicking >). The pre-expansion preview is raw JSON. A chip + key list would replace that without breaking the expansion affordance.

Three rendering directions (compact reference)

All three variants compose with the existing expansion model — they don't replace it. The chip + preview is what the user sees in a collapsed cell. Clicking > on the column header still expands homogeneous nested objects into sub-columns as today. The variants below differ only in how the collapsed leaf cell communicates its shape.

A. Chip + count + first keys★ recommended

Two compact lines per cell. Chip + element count + first 2-3 sample keys/values.

Kiribati, Fiji

Tradeoffs
Pros: scannable, type-explicit, predictable height.
Cons: two-line cells make rows variable height (mitigated by row-height setting).

Compact, syntax-highlighted, truncated. Same row height as string cells.

Tradeoffs
Pros: uniform row height, dense, easy bulk-scanning.
Cons: no chip = type signal is purely visual (color, braces). Less explicit. Truncation hides nested structure.
