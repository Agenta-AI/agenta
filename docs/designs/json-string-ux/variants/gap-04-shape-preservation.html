Gap 04 — Union-projected fields aren't visually distinct from authored fields

First correctness gap. After applying chips (gap-01) to the table (gap-02) and drill-in (gap-03), the next question is: what do those chips say about fields that aren't actually authored?

State stays clean unless the user explicitly edits the JSON view. The "pollution" in the screenshots is a render-time projection at EntityDualViewEditor.tsx:144–155 (the ?? "" fallback). The column union is a convenience feature — if testcase A has columns X+Z and testcase B has only X, the user can author Z in B without manually adding the column.

The remaining concerns are about identity ambiguity: the user can't tell authored-empty from union-projected-missing.

Anchor components: EntityDualViewEditor (the ?? "" projection at the JSON-view boundary) and DrillInFieldHeader (where a "not authored" chip stacks on top of the type chip). Aware of the drift: valueMode="string" (testcase) and valueMode="native" (trace) auto-parse stringified JSON differently — we should not extend the projection across that boundary without thinking through what the union means for trace data.

Three risks (where the projection actually causes problems)

RiskWhat happensImpact
1. User confusion An empty "" cell could mean "user authored empty string" or "column projected from another row." Both render identically. Medium · review/debug clarity
2. JSON-view edits handleJsonChange dispatches the parsed object verbatim. Any keystroke replays the projection (with empties) into the draft. High · only path that pollutes BE
3. Template-runtime shadowing If the template resolver receives the projection (not data), {{geo.region}} resolves to "" instead of falling back to nested traversal. Depends on resolver path · verify

Three approaches to the fix

All three keep the convenience feature. They differ in how the projection is visually marked and how the JSON-edit path is bounded.

A. Distinct projected fields + scoped JSON save★ recommended

Keep the convenience feature. Mark projected fields visually so users can tell. JSON-edit path diffs against actual data so the projection can't accidentally persist.

▾ country str
▾ geo obj
{ "region": "Micronesia", "subregion": "Central Pacific" }
▾ geo.region not authored
▾ user.profile.name not authored

JSON view default: as authored (BE shape). Toggle reveals projection. JSON-edit handler diffs against actual data before dispatching.

Tradeoffs
Pros: keeps convenience feature. User can distinguish authored vs. projected. JSON-edit path bounded.
Cons: new UI vocabulary (dashed border + chip). One toggle in JSON view.

B. Render only authored keys

Strict shape preservation. Loses the convenience feature.

▾ country str
▾ geo obj
{ "region": "Micronesia", "subregion": "Central Pacific" }
▾ correct_answer str
+ Add field that exists in other rows: geo.region, user.profile.name, …
Tradeoffs
Pros: smallest diff, 1:1 with BE.
Cons: loses the convenience feature. Users must click "+ add" each time they want to author a missing column.

C. Status quo + fix Risk 2 only

Keep the rendering as-is. Filter the JSON-edit path so it can't replay the projection into storage.

Display: unchanged. Drill-in Fields and JSON views show the union projection as today.

handleJsonChange: parse → diff against testcase's actual data → drop empty-string fallbacks for keys that weren't in data → dispatch the filtered diff.

User-facing behavior: identical for unedited testcases. JSON-view edits no longer accidentally persist projected empties.
Tradeoffs
Pros: minimal change. Eliminates the storage-pollution risk.
Cons: doesn't address Risk 1 (user confusion). Doesn't help template-runtime if Risk 3 is real.
Recommendation: Variant A. Preserves the convenience feature (which is the point of the union model) while making the projection visible and bounding the JSON-edit path so it can't accidentally persist projected empties. The dashed-border + not authored chip is also useful for template-authoring — when the user runs the playground on a single testcase, they can immediately see which referenced variables aren't authored on that row.

Verification still worth doing

Confirm Risk 3 (template-runtime) by tracing what shape the playground / evaluation runner passes to the template resolver. If the resolver receives entityData directly, Risk 3 is hypothetical. If it gets the rendered JSON-view string or a synthesized projection, the empty-literal shadowing is real and Variant A's "as authored" default needs to extend to the runtime path too.
