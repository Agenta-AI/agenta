Gap 06 — Messages renderer coverage is incomplete

Polish layer, but also where the drill-in drift (index § drift) is most user-visible. After chips (gap-01), table cells (gap-02), drill-in root view (gap-03), and the two correctness gaps (gap-04, gap-05), one specialized renderer is still inconsistent across the drill-in family.

ChatMessageEditor auto-detection lives only in DrillInContent (DrillInContent.tsx:1284, isChatMessageObject check). BeautifiedJsonView unwraps message envelopes but renders cleaned JSON, not the editor. JsonEditorWithLocalState shows raw JSON with no message detection. TraceSpanDrillInView's two render paths (rootScope="span" beautified vs rootScope="attributes" field-drill) pick differently. Same message object, three different views depending on which path caught it.

Anchor decision: either lift isChatMessageObject + ChatMessageEditor into a shared renderer that BeautifiedJsonView and JsonEditorWithLocalState both delegate to, or accept that the three views are alternative renderings and document why. Don't ship a fix that lands in one and leaves the other two stale.

Today Current state

ChatMessageEditor exists and works in two of four surfaces. tool_calls never gets the rich treatment.

Fixture 07 testset table: messages column ✓ uses ChatMessageEditor (system/user/assistant cards). tool_calls column ✗ shows raw JSON [] or single-line summary.
Drill-in Fields view (root): even messages ✗ falls through to a raw JSON code editor. The dedicated renderer doesn't kick in until drill-in.
Drill-in (drilled in): messages ✓ ChatMessageEditor (Assistant/Tool cards, drill-in arrows). tool_calls ✗ still raw JSON code editor with the function args nested as escaped strings.

Detection works (isChatMessageObject in fieldUtils.ts:185). The gap is wiring detection → renderer at the right surfaces.

User reads the entire trace at a glance without drilling.

Three approaches

A. Wire dedicated renderers everywhere★ recommended for messages

If detectDataType === "messages" → ChatMessageEditor. Else fall through to chip + preview from gap-02.

Drill-in Fields view (root) with messages array:

▾ messages msgs 3 items Drill In ▸
SYSTEM
You are a geography expert. Answer concisely.
USER
What is the capital of Tuvalu?
ASSISTANT
The capital of Tuvalu is Funafuti.
Tradeoffs
Pros: consistent renderer everywhere. Users get rich view for the types we have rich renderers for.
Cons: more code paths to maintain.

B. Add tool_calls as a detected type

Extend detectDataType to recognize tool-call shape. Wire to existing ToolCallView component.

Drill-in (drilled in) with tool_calls:

▾ tool_calls tool 1 call
function lookup_country call_abc123
arguments:
{ "country": "Kiribati" }
Tradeoffs
Pros: keeps detection central. New tool-call chip adds clarity. Reuses existing ToolCallView.
Cons: heuristic detection (tool_calls shape varies). False positives possible.

C. Defer tool_calls to v2

Wire only messages everywhere. Leave tool_calls as raw JSON until users complain.

Drill-in (drilled in) with tool_calls:

▾ tool_calls arr 1 record
[
  {
    "id": "call_abc123",
    "type": "function",
    "function": { "name": "lookup_country", ... }
  }
]
Tradeoffs
Pros: smallest change. No false-positive risk.
Cons: users have zero affordance to read tool calls at a glance.
Recommendation: A for messages, C for tool_calls in v1. Messages detection is unambiguous, the renderer is polished, ship the wiring. Tool_calls detection is heuristic — defer until real-world traces show users struggling. Don't lock in a renderer choice based on one fixture.
