JSON ↔ String UX — Design Variants

Three variants per pattern, generated as concrete reactions to the recommendations in docs/designs/json-string-ux/. The team picks a direction per pattern; this is the artifact for the design conversation.

Existing capabilities (`detectDataType`, round-trip preservation, transparent stringified-JSON traversal) are not changed by any of these variants. They visualize the missing pieces only: visible chips, table cell rendering, conversion affordance, variables panel.

A. Type indicator chip placement

drawer Fields view, variable panel, observability
Where and when to show a type chip. The user needs to answer "is this a string or an object?" without clicking.
My recommendation: A3 — Ambiguous-only. Strings get no chip in compact rows; objects/arrays/null/messages always show. Cleaner default, still meets RFC requirement.

A1. Always-on

Every value, every row gets a chip. Maximum signal, but every cell carries chip chrome.

Property values

agenum42

sessionmsgs[3 messages ▸]

Tradeoffs
Pros: consistent visual language. Type info always visible. Mahmoud's RFC literal reading: "show wherever values are edited or inspected."

Cons: chip chrome on every row adds visual weight. Strings (most common) get a chip that says nothing the quotes don't already say.

A2. Hover-only

Clean default. Chips appear when the user hovers a row, plus a tooltip with type detail.

Property values (hover row 2 to preview)

profile obj { name: "Ada", … } Object · 3 properties
name, tags, lastLogin · Click to convert

session[3 messages ▸]
Tradeoffs
Pros: minimal visual noise at rest. Reveals on intent. Mobile-hostile but most authoring is desktop.

Cons: discoverability — users might never hover and never learn types are detected. Mobile users see nothing.

A3. Ambiguous-only ★ recommended

Strings get no chip (rendering is self-evident: quotes). Objects/arrays/null/messages always show.

Property values
country"Tuvalu"
profileobj{ name: "Ada", … }
age42
activetrue
tagsarr[ "x", "y", … ]
sessionmsgs[3 messages ▸]
notesnullnull
Tradeoffs
Pros: clean for the common case (strings). Important types stay visible. Honest about which signal needs chrome and which doesn't.

Cons: requires "is rendering self-evident?" judgement. RFC's literal reading is "always-on" — this might need Mahmoud's blessing.

B. Testset table — object/array cell rendering

testset table page
Cells are narrow (~200px). Object cells today flatten to truncated text. How do we signal "this is an object" and preview shape?
My recommendation: B1 — Chip + count + key list. Compact, scannable, matches the rest of the chip system.

B1. Chip + count + key list ★ recommended

[obj] { 2 } countryName, capital — uses ~1.5 lines. Scannable type + shape preview.

correct_answercountryoutputs
"The capital of Tuvalu…" "Tuvalu" "The capital of Tuv…"
"The capital of Kiribati is Tarawa." "Kiribati"
obj { 2 }
countryName, capital
"What capitals…" "Spain"
arr [ 5 ]
"Madrid", "Barcelona"…
Tradeoffs
Pros: scannable, type-explicit, predictable height.
Cons: two-line cells make rows variable height.

B2. Mini JSON tree

Vertical 2-3 line indented preview. More structural, more vertical space.

correct_answercountryoutputs
"The capital of Tuvalu…" "Tuvalu" "The capital of Tuv…"
"The capital of Kiribati is Tarawa." "Kiribati"
{
  "countryName": "Kiribati",
  "capital": "S. Tarawa"
}
"What capitals…" "Spain"
[
  "Madrid",
  "Barcelona", …
]
Tradeoffs
Pros: structural, syntax-highlighted, faithful to JSON.
Cons: uses ~5 lines per object cell. Rows become very uneven.

B3. Inline single-line JSON

Compact single line with syntax highlighting, truncated. Same row height as strings.

correct_answercountryoutputs
"The capital of Tuvalu…" "Tuvalu" "The capital of Tuv…"
"The capital of Kiribati is Tarawa." "Kiribati" { "countryName": "Kiribati", "capital": "South Ta…" }
"What capitals…" "Spain" [ "Madrid", "Barcelona", … ]
Tradeoffs
Pros: uniform row height, dense, easy bulk-scanning. Syntax highlighting differentiates from string cells.
Cons: no chip = type signal is purely visual (color, braces). Less explicit.

C. Drawer — per-property conversion affordance

testcase drawer Fields view
Where and how to expose the "convert string → object" action. Reminder: by default the system preserves storage — conversion is an opt-in user action, not a default behavior.
My recommendation: C1 — Inline buttons. Discoverable, single click. Optional layered chip-dropdown for power users.

C1. Inline buttons ★ recommended

[→ obj] [→ arr] buttons next to property header. One click conversion.

▾ profile str
→ obj → arr
{"name": "Ada"}
▾ country str
→ obj → arr
Tuvalu
▾ tags arr
→ str
[ "alpha", "beta" ]
Tradeoffs
Pros: discoverable, one-click. Always visible.
Cons: adds chrome on every property header. Visually busy on long forms.

C2. Chip dropdown

Click the type chip to open a conversion menu. Compact, but discoverability requires hover.

▾ profile str ▾
{"name": "Ada"}
● String current
→ Convert to object
→ Convert to array
→ Convert to number
▾ country str ▾
Tuvalu
▾ tags arr ▾
[ "alpha", "beta" ]
Tradeoffs
Pros: compact. Type chip itself is the action. Scales to many type options without crowding.
Cons: not obvious the chip is clickable. Requires hover/click discovery.

C3. Three-dot menu

Conventional ⋯ menu next to header opens context menu with conversions + delete.

▾ profile str
⋯
{"name": "Ada"}
→ Convert to object
→ Convert to array
→ Convert to number
⎘ Duplicate
🗑 Delete
▾ country str
⋯
Tuvalu
Tradeoffs
Pros: conventional pattern (⋯ = "more"). Groups conversion with delete/duplicate. Less header chrome.
Cons: two clicks (open menu → pick action). Conversion buried with destructive actions.

D. Playground variables panel

playground right side
Show what's referenced in the prompt + what's available from testcase/trace/evaluator context. User clicks a variable to insert at cursor.
My recommendation: D1 — Sectioned panel. Source-grouped, expandable nesting. Most legible when many sources are active.

D1. Sectioned panel ★ recommended

Sections by source, collapsible. Each section has its own header and contents.

▾ Discovered from prompt
{{inputs}}objtestcase
{{outputs}}objupstream
▾ Available · testcase
▾inputsobj
countrystr
correct_answerstr
▾outputsobj
countryNamestr
capitalstr
▸ Available · evaluator 2 vars
Tradeoffs
Pros: source is unambiguous, scales when many sections are populated, easy to collapse what's not relevant.
Cons: section chrome eats vertical space when sections are sparse.

D2. Flat list + filter chips

All variables in one scrollable list. Source filter chips at top. Maximum density.

all · 8 testcase · 4 upstream · 2 evaluator · 2
{{inputs}}objtc
{{outputs}}objup
countrystrtc
correct_answerstrtc
countryNamestrup
capitalstrup
correct_answer_keystreval
thresholdnumeval
Tradeoffs
Pros: highest density. Filter is fast. Search-friendly.
Cons: source distinction relies on tiny tags (`tc`/`up`/`eval`) which take learning. Loses the prompt-discovered grouping.

D3. Tabbed (Discovered | Available)

Two tabs separate prompt-referenced from context-available. Inside each, secondary grouping.

Discovered 2 Available 8
{{inputs}}objresolves: testcase
{{outputs}}objresolves: upstream
All prompt-referenced variables resolve. Switch to Available to insert more.
Tradeoffs
Pros: clear separation between "what the prompt uses now" and "what else is around." Discovered tab can show resolution status (resolves/missing).
Cons: tab switching costs context. Easy to forget the other tab exists.
