## Evaluation Runs

An evaluation run is the table definition. It stores status, flags, metadata, references, and `data`.

`EvaluationRunData` contains:

- `steps`: the logical columns in the evaluation table.
- `repeats`: the z dimension for repeated invocations or annotations.
- `mappings`: labels and value paths that connect UI columns to step outputs.

Each step has:

- `key`
- `type`: `input`, `invocation`, or `annotation`
- `origin`: `custom`, `human`, or `auto`
- `references`: entity references such as evaluator, evaluator revision, workflow, query, or testset references.
- optional `inputs`

The run query endpoint is:

- `POST /api/evaluations/runs/query`

Supported run filters today are mostly identity, status, flags, tags, and references. There is no run query support for filtering by scenario-cell values.

## Evaluation Scenarios

A scenario is the row entity. It is independent from result and metric payloads.

Important fields:

- `id`
- `status`
- `interval`
- `timestamp`
- `run_id`

The scenario query endpoint is:

- `POST /api/evaluations/scenarios/query`

Supported scenario filters today are:

- `ids`
- `run_id` / `run_ids`
- `status` / `statuses`
- `interval` / `intervals`
- `timestamp` / `timestamps`
- `flags`
- `tags`

The DAO applies cursor windowing through `apply_windowing` over UUID7 `id` in ascending order for normal evaluation views. The web table uses descending cursor order for online evaluations.

## Evaluation Results

A result is the cell execution reference for one scenario, one step, and one repeat.

Important fields:

- `hash_id`
- `trace_id`
- `testcase_id`
- `error`
- `status`
- `interval`
- `timestamp`
- `repeat_idx`
- `step_key`
- `scenario_id`
- `run_id`

The database enforces uniqueness on:

```text
project_id, run_id, scenario_id, step_key, repeat_idx
```

The result query endpoint is:

- `POST /api/evaluations/results/query`

Supported result filters today are:

- `ids`
- `run_id` / `run_ids`
- `scenario_id` / `scenario_ids`
- `step_key` / `step_keys`
- `repeat_idx` / `repeat_idxs`
- `timestamp` / `timestamps`
- `interval` / `intervals`
- `status` / `statuses`
- `flags`
- `tags`

Results deliberately do not carry the output values. They carry references to traces or test cases, plus errors and state.

## Evaluation Metrics

Metrics are the materialized value surface currently available inside evaluation storage.

Important fields:

- `status`
- `data`
- `interval`
- `timestamp`
- `scenario_id`
- `run_id`

The metrics table has partial uniqueness rules:

- Global metrics: unique by `project_id, run_id` where `scenario_id IS NULL AND timestamp IS NULL`.
- Variational metrics: unique by `project_id, run_id, scenario_id` where `timestamp IS NULL`.
- Temporal metrics: unique by `project_id, run_id, timestamp` where `scenario_id IS NULL`.

The metrics query endpoint is:

- `POST /api/evaluations/metrics/query`

Supported metric filters today are:

- `ids`
- `run_id` / `run_ids`
- `scenario_id` / `scenario_ids`
- `timestamp` / `timestamps`
- `interval` / `intervals`
- `status` / `statuses`
- `flags`
- `tags`

`scenario_ids` and `timestamps` accept booleans in addition to lists:

- `scenario_ids: false` means run-level metrics.
- `scenario_ids: true` means scenario-level metrics.
- `timestamps: false` means non-temporal metrics.
- `timestamps: true` means temporal metrics.

There is currently no filtering by `data` path or computed metric value.

## Metric Production

Metrics are generated by `EvaluationsService.refresh_metrics`.

Refreshable run steps are:

```python
METRICS_STEP_TYPES = {"invocation", "annotation"}
```

Default metrics are collected from trace attributes:

- `attributes.ag.metrics.duration.cumulative`
- `attributes.ag.metrics.errors.cumulative`
- `attributes.ag.metrics.costs.cumulative.total`
- `attributes.ag.metrics.tokens.cumulative.total`

For annotation steps, evaluator output schemas are converted into metric specs by `get_metrics_keys_from_schema`.

Schema conversion currently maps:

- `boolean` to `binary`
- `string` with enum to `categorical/single`
- plain `string` to `string`
- `number` to `numeric/continuous`
- `integer` to `numeric/discrete`
- array of enum strings to `categorical/multiple`
- object without properties to `json`

Metric data is stored as:

```text
evaluation_metrics.data[step_key][metric_path] = analytics bucket metric
```

For evaluator outputs, the metric path is prefixed with:

```text
attributes.ag.data.outputs.
```

The exact leaf shape is analytics-dependent. The frontend already handles several shapes, including scalar values, stats objects, frequency distributions, and a legacy `{value: ...}` leaf shape.

## Frontend Data Flow

The run detail page loads the run and derives columns from `run.data.steps` and `run.data.mappings`.

Scenario rows are loaded through a virtualized infinite table:

- `evaluationPreviewTableStore` requests windows of scenarios.
- `fetchEvaluationScenarioWindow` calls `POST /evaluations/scenarios/query`.
- Rows are skeletonized while windows load.
- The frontend never has all scenario rows for large runs.

For visible or otherwise subscribed rows, the frontend batches result requests:

- `scenarioStepsBatcherFamily` collects scenario IDs.
- It calls `POST /evaluations/results/query` with `run_id`, `run_ids`, and `scenario_ids`.
- Results are grouped by `scenarioId`.

For visible metric cells, the frontend separately batches metric requests:

- `evaluationMetricBatcherFamily` collects scenario IDs.
- It calls `POST /evaluations/metrics/query` with `scenario_ids`.
- It intentionally does not constrain by `run_ids` in the current scenario-metric batcher, to avoid over-filtering.
- Metric rows are merged into `{raw, flat}` maps per scenario.
- Cell lookup uses step key, evaluator key, metric key, canonicalized path, suffix matching, and raw nested lookup.

Run-level metrics are fetched separately with:

```json
{
  "metrics": {
    "run_ids": ["..."],
    "scenario_ids": false,
    "timestamps": false
  }
}
```

## Consequence for Filtering

Filtering cannot be correct in the frontend for large runs because:

- The frontend has only a window of scenarios.
- Results and metrics are fetched lazily by visible scenario IDs.
- Result rows do not contain raw output values.
- Metric rows contain the closest existing materialization of filterable values, but only for fields that metric refresh decides to compute.

Backend filtering has to return scenario windows after applying the predicate. The virtualized table should continue paging scenarios, but the scenario query must accept filter criteria and produce only matching scenario IDs.

`evaluation_metrics.data` is JSONB via `DataDBA`, but there is no explicit index on it in `EvaluationMetricsDBE`.

Filtering over metric values will need either expression indexes for hot paths or a separate normalized metric-value index table if broad ad hoc filtering becomes a performance requirement.

## Tracing Query and Analytics Surface

There is an existing tracing filter language in `api/oss/src/core/tracing/dtos.py`.

The core pieces are:

- `TracingQuery`
- `Filtering`
- `Condition`
- `MetricSpec`

Supported filter fields include:

- `trace_id`
- `trace_type`
- `span_id`
- `span_type`
- `parent_id`
- `span_name`
- `span_kind`
- `start_time`
- `end_time`
- `status_code`
- `status_message`
- `attributes`
- `events`
- `links`
- `references`
- `hashes`
- lifecycle fields
- `content`

Supported operators include numeric comparisons, string operators, list operators, dictionary containment operators, and existence operators.

Relevant endpoints:

- `POST /spans/query`
- `POST /traces/query`
- `POST /spans/analytics`

`/spans/query` can apply conditions directly to span attributes. This is the best existing primitive for "find traces whose underlying raw span data satisfies a predicate".

Analytics accepts a `TracingQuery` plus `MetricSpec[]`. It applies the same filtering first, then bucketizes matched root spans and computes metric summaries. The implementation currently:

- defaults analytics to root spans through `parent_id IS NULL`
- removes the leading `attributes.` prefix from `MetricSpec.path`
- extracts values from `SpanDBE.attributes`
- returns `MetricsBucket` entries keyed by timestamp and metric path

This is the same mechanism used by `EvaluationsService.refresh_metrics` to materialize `evaluation_metrics.data`.

Important limitation: analytics returns aggregate buckets, not the scenario IDs or trace IDs that matched a predicate. It can answer "what are the stats over these filtered traces?" but not directly "which scenario rows should stay visible?" without an additional trace-to-scenario lookup.

## Trace-to-Scenario Mapping

Evaluation results already provide the bridge from scenarios to traces:

```text
evaluation_results(run_id, scenario_id, step_key, repeat_idx, trace_id)
```

A trace-backed scenario filter can therefore be implemented by:

1. Selecting candidate result rows for a run, step key, and repeat policy.
2. Applying a tracing `Filtering` predicate over the candidate `trace_id` set.
3. Mapping matched trace IDs back to `scenario_id`.
4. Paging `evaluation_scenarios` over those matching scenario IDs.

The inverse is also possible:

1. Load scenario IDs for the current window or candidate set.
2. Load their result trace IDs.
3. Query spans for those trace IDs with extra conditions.
4. Keep the scenarios whose trace IDs matched.

The second approach is not enough for table-wide filtering because it only filters the loaded window. A backend implementation must apply the trace predicate before scenario windowing.

## Metric Filtering vs Trace Filtering

Metric filtering and trace filtering overlap but are not identical.

Metric filtering operates over materialized `evaluation_metrics.data`. It is aligned with what metric cells show, including aggregate stats produced by analytics.

Trace filtering operates over raw span attributes. It can support conditions that are not materialized as metrics, but it must still map matched traces back to scenarios.

Analytics can recompute metric summaries over filtered traces, but by itself it does not solve scenario-row filtering because it returns aggregate buckets rather than per-scenario row membership.
