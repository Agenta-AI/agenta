# Clean up meters — generalize scope/period, prepare global read limits

## What this unlocks

This PR turns the `meters` table from "one quota row per organization per month" into a general scope × period meter store. Any quota in the catalog can now declare a `scope` (organization / workspace / project / user) and a `period` (daily / monthly / yearly), and the meters layer stores one row per `(scope, period, key)` keyed by a deterministic `meter_id`.

The immediate capability this lights up is **per-user, per-day tracing read limits** (`Counter.TRACES_RETRIEVED`), declared structurally on every plan with `limit=None` for now — the wiring is in place, an operator can flip on a real limit per plan without any code change. Counter names also pick up an explicit verb (`TRACES_INGESTED`, `TRACES_RETRIEVED`, `CREDITS_CONSUMED`, `EVALUATIONS_RUN`) so the call site tells you which direction data is moving.

`/billing/usage` exposes `period` and `scope` additively, and reads **per-caller**: for each quota, the response returns the single meter row matching the caller's ambient `AuthScope` projected to `quota.scope`'s granularity (same projection `check_entitlements` uses, so numerator and denominator stay at the same scope). The route has no `organization_id` path/query/wrapper param — identity comes from the ambient context. The frontend usage card renders the new fields. Stripe reporting is untouched — `TRACES_RETRIEVED` is not in the `REPORTS` allowlist, so it's tracked and surfaced internally and to customers but not billed.

The auth path now publishes a typed `AuthContext` on a `ContextVar` (organization / workspace / project / user IDs plus credentials), which removes the need to keep threading `Request` through services just to read those values, and lets entitlement checks default to the ambient caller's scope at whatever granularity the quota declares.

The catalog cleanup drops two dead `Counter` values (`EVALUATORS`, `ANNOTATIONS`), one dead `Gauge` (`APPLICATIONS`), and `Flag.HOOKS`. The previously never-incremented `EVALUATIONS` (only rolled back on failure, never charged) is renamed to `EVALUATIONS_RUN` and given proper `+1` / `-1` wiring at the three evaluation-run creation handlers (broad refund on any exception, plus a shortfall refund on the silent-return path so charges match actual creations). `TRACES_RETRIEVED` hard-check fires after every span/trace fetch or query (analytics, sessions, and users endpoints are deliberately excluded — they return aggregates or IDs, not trace data); declared `strict=True` everywhere, so the request that would cross a real limit is itself rejected.

## What changed under the hood

**`meters` schema.** Columns added: `meter_id UUID` (new PK), `workspace_id`, `project_id`, `user_id` (UUID, nullable), `day SMALLINT` (nullable). `year` and `month` relaxed to nullable, the `(0, 0)` gauge sentinel promoted to real `NULL`s. The legacy composite PK is rebuilt as a non-unique secondary index extended with `day`, so the org-rollup scan path that `/billing/usage` and the entitlements soft-check use stays index-supported. `MeterDBE` keeps the FK to `subscriptions.organization_id` and nothing else — meters can legitimately outlive the workspaces/projects/users they were scoped to.

**`meter_id` canonicalization.** A single function `compute_meter_id(*, scope: MeterScope, period: MeterPeriod, key)` in `ee.src.core.meters.types` is the only thing that produces a `meter_id`. The DAO, helpers, and the Alembic backfill all go through it; the database has no SQL-side mirror. The canonical string excludes `None` fields, uses short alphabetically-sorted labels (`org`, `wrk`, `prj`, `usr`, `y`, `m`, `d`, `key`), and is hashed against a namespace UUID derived from a literal `"agenta"` string (`uuid5(NAMESPACE_DNS, "agenta")` lives in `env.agenta.uuid_namespace`) so the IDs survive auth-key rotations. `MeterScope` validates the scope hierarchy, `MeterPeriod` validates the calendar (Feb 30 is rejected), and `MeterDTO` constructs both internally so any invalid shape fails at DTO construction. The `MeterDTO` validator also enforces single-source-of-truth at write time: it always recomputes the canonical `meter_id` from `(scope, period, key)`, and if a caller supplied a different value the module-level logger warns and the canonical value wins — no silent override, no way to bypass canonicalization through the constructor either.

**Migration.** One transactional Alembic revision (`9d3e8f0a1b2c_reshape_meters_table`) does the enum reshape and the schema reshape together. The Postgres `meters_type` enum is type-swapped (not `ALTER TYPE ADD VALUE`) so renames, additions, and removals happen atomically: `APPLICATIONS` rows are deleted, `TRACES`/`CREDITS`/`EVALUATIONS` rows are mapped to their verb-explicit successors via a `CASE` in the `USING` clause, `TRACES_RETRIEVED` is added. The legacy composite PK is dropped before `year`/`month` are relaxed (Postgres refuses `DROP NOT NULL` on a PK column); `organization_id` is relaxed to NULL in the same step so the schema matches the ORM, which now declares it nullable. `meter_id` is then backfilled by importing the same canonicalizer the runtime uses — deliberately, as a documented trust-model choice: dual implementations drift, and any future change to the canonical form requires a re-backfill migration that would move both sides together. Downgrade is symmetric and lossy: before dropping the new columns it deletes every row whose identity depends on `workspace_id` / `project_id` / `user_id` / `day` (the old schema has no representation for finer scopes or daily granularity, so the legacy PK rebuild would otherwise collapse distinct rows and crash with a duplicate-key violation). It then drops the new PK + index + columns, deletes any `organization_id IS NULL` rows (same roundtrip story), restores `organization_id` and `year`/`month` to `NOT NULL DEFAULT 0`, reverses the enum swap, and recreates the legacy composite PK last. The historical `7990f1e12f47_create_free_plans` migration was updated to use raw SQL (single `text` import hoisted to module scope) with `_LEGACY_APPLICATIONS_KEY` and `_LEGACY_USERS_KEY` string constants and `CAST(:key AS meters_type)` so both legacy USERS and APPLICATIONS meter blocks replay cleanly on a fresh database without depending on the (now-removed) Python enum members.

**`Quota` and `Probe` reshape.** `Quota.monthly: bool` is replaced by `Quota.period: Optional[Period]` (where `Period` is the new `DAILY` / `MONTHLY` / `YEARLY` enum); `Quota.scope: Optional[Scope]` is added with the convention that `None` means `ORGANIZATION`. `Quota.strict` and `Probe.delta` move from `Optional[bool] = False` to `Optional[bool] = None` (None means False), and the previous minute-valued `Periods(str, Enum)` is renamed to `Retention(int, Enum)` so `Quota.retention: Optional[Retention]` flows straight through. `Probe.monthly` becomes `Probe.period`. Every existing plan declaration is migrated mechanically.

**DAO layer.** `MetersDAO.dump` / `bump` / `fetch` / `check` / `adjust` all key on `meter_id`. `adjust` uses `ON CONFLICT (meter_id) DO UPDATE` — single-column conflict target instead of the old four-column tuple. The upsert is gated by a two-step check. First a Python-side fast-path rejects any *predictable self-overshoot* in both modes: absolute writes where `meter.value > limit` and delta writes where `meter.delta > limit` short-circuit to `(False, ...)` with no DB call. Then a `WHERE` predicate branches on `quota.strict`: strict emits `greatest(value + delta, 0) <= limit` so the request that would cross the limit is itself rejected; non-strict emits `value < limit` (delta path) — letting one request cross the line from below while denying any row that is already at-or-over the limit. The shared `delta <= limit` rule (enforced by the Python fast-path) is what prevents a single huge delta from sneaking through non-strict's permissive SQL clause. `check` is always strict (`current + delta <= limit`). `fetch(*, scope: Optional[MeterScope], key: Optional[Meters], period: Optional[MeterPeriod])` distinguishes two filter modes per family: when the parameter itself is `None`, no filter is applied on that family (the "rollup" escape hatch); when a structured `MeterScope` / `MeterPeriod` object is supplied, *every* dimension is bound uniformly — `None` dimensions translate to `column IS NULL`, not "any value", because a `None` dimension is part of the canonical meter identity ("not applicable at this grain"). Without that, an org-scoped/monthly read would also match finer-scoped or DAILY rows for the same `(org, key, year, month)`. The "all meters" admin escape hatch is `scope=None` (no scope filter at all), not a `MeterScope()` with every dimension `None`. The same method serves org-rollup scans (`scope_from(organization_id=...)`) and pinpoint reads (`scope_from(scope=Scope.USER)`) without branching. `key` is typed as `Meters` (not `str` / `Counter` / `Gauge`) — the DB column is `SQLEnum(Meters, name="meters_type")` with name-binding (uppercase Python enum names), so callers holding a `Counter`/`Gauge` convert at the boundary with `Meters[key.name]`. The type contract pins the DB binding contract; without it, passing a `Counter` would bind the lowercase `.value` and silently miss every row. `fetch` also eagerly loads the optional `subscription` relationship via `joinedload(MeterDBE.subscription)` — `_dbe_to_dto` reads `meter.subscription` after the session closes, so a lazy-load there raises `MissingGreenlet` and 500s `/billing/usage`. Its docstring carries a WARNING that passing `scope=None` returns every meter row in the table; the three current callers (`MetersService.fetch` passthrough, `BillingRouter.fetch_usage`, the `check_entitlements` soft-check path) all pass a tenant-bound scope, and the warning instructs future contributors to audit before broadening the surface. Period normalization happens via a `with_period` helper on `MeterDTO` that returns a new instance with a recomputed `meter_id` rather than mutating in place. `_normalize_period_on_meter` only snaps to the current quota bucket when the caller hasn't already populated the meter's `year`/`month`/`day` — an explicit period (e.g. a backdated adjustment) flows end-to-end through the cache key, the DTO's `meter_id`, and the DB upsert without rewrite. The old `compute_billing_period` was replaced by two helpers in `ee.src.utils.entitlements`: `monthly_period_from` (honors Stripe-style anchor day) and `period_from` (builds a `MeterPeriod` for the current moment at the requested granularity).

**Entitlements helper.** `check_entitlements(*, key, delta=None, cache=False, scope=None, period=None)` is the public surface. When `scope`/`period` are omitted, the helper projects the ambient `AuthScope` (from the new `ContextVar`) down to the granularity declared by `quota.scope` via `scope_from(scope=quota.scope)` — `quota.scope=None` maps to organization-only, which is the common case (the unified `scope_from` treats `scope=None` as `Scope.ORGANIZATION`). Period defaults come from `period_from(period=quota.period, anchor=...)`. Cache keys are scope+period+key projections rather than the previous `(organization_id, key, year, month)` tuple. The cache-mode preflight mirrors the DAO's strict/non-strict predicate exactly — strict uses `current + delta <= limit`, non-strict uses `delta <= limit and current < limit` — so Layer 1 is never stricter than Layer 2 (the authoritative DAO upsert). The error policy is explicit: `EntitlementsException` (config bugs — invalid key, missing plan, no subscription) propagates; everything else fails open, so a meter glitch never blocks a request. The soft-check DB fallback (cache miss path) converts the inbound `Counter`/`Gauge` to a `Meters` member by name (`Meters[key.name]`) before calling `MetersDAO.fetch`, matching the column's name-binding (uppercase Python enum names). `MetersService` and `SubscriptionsService` are wired via two helpers in the entitlements module: `register_entitlements_services(...)` (the raw setter) and `bootstrap_entitlements_services(*, meters_service=None, subscriptions_service=None)` (builds defaults from DAOs when no services are passed). The EE HTTP entrypoint passes already-built instances to share with `BillingRouter`; every worker entrypoint (`worker_tracing`, `worker_events`, `worker_evaluations`, `worker_webhooks`) gates both the import and the call on `if is_ee():` and invokes the no-arg form after env validation, so `check_entitlements` works in EE worker processes and OSS-only builds (where the `ee.*` package isn't on the path) still start cleanly. Without that bootstrap step the authoritative `TRACES_INGESTED` hard-check in the async tracing worker and the `ACCESS` gate in the events worker would silently fail open.

**Write-side enforcement (`EVALUATIONS_RUN`).** Three handlers — `EvaluationsRouter.create_runs`, `SimpleEvaluationsRouter.create_evaluation`, `SimpleQueuesRouter.create_simple_queue` — check the quota after the permission check, return `429 Too Many Requests` on `not allowed`, refund with `delta=-N` if the service call raises (any `Exception` — broad-safety trade-off), and refund the *shortfall* between charged `N` and actual creations on the silent-return path (`create_runs` refunds `N - len(runs)`, the single-create handlers refund `1` when the service returned `None`). The shortfall refund covers the DAO's `suppress_exceptions(default=[])` decorator and the simple services' early `return None` guards — both swallow what would otherwise be an exception. No domain exception or HTTP exception subclass was introduced; the router translates the boolean directly.

**Read-side enforcement (`TRACES_RETRIEVED`).** Every trace/span retrieval handler captures the result of `check_entitlements(key=Counter.TRACES_RETRIEVED, delta=...)` and raises `HTTPException(429, "You have reached your trace retrieval quota for this period.")` when `allowed is False` — eight sites total: `query_spans`, `fetch_spans`, `fetch_span` on `SpansRouter`; `query_traces`, `fetch_traces`, `fetch_trace` on `TracesRouter`; plus `query_spans` and `fetch_trace` on the deprecated `TracingRouter` so the legacy `/tracing/*` mounts are covered too. No `cache=` kwarg — `cache=False` is the default, and reads have no async worker for a second-pass adjust, so each handler does a single synchronous hard-check that atomically upserts the meter row via `MetersDAO.adjust()`. (Compare with `TRACES_INGESTED`, which keeps `cache=True` at the ingestion gates because the async tracing worker performs the authoritative hard-check downstream.) Delta is `len(traces)` for trace-shaped responses, `len({s.trace_id for s in spans})` for span-shaped responses, `1` / `0` for singletons. Every plan declares `TRACES_RETRIEVED` with `strict=True` so the DAO upsert's `WHERE` clause is `greatest(value + delta, 0) <= limit` — the request that would push the meter past the limit is itself rejected; with a 10-trace limit, the 11th retrieval gets the 429, not the 12th or 13th. Non-strict mode (`value < limit` plus the shared `delta <= limit` rule) would have permitted the one cross-the-line request from below, but `TRACES_RETRIEVED` deliberately opts out. Analytics, sessions, and users endpoints are explicitly excluded — they return aggregates or IDs, not trace data.

**Auth + context wiring.** A new `AuthContext` Pydantic model (credentials + `AuthScope`) is published on a `ContextVar` by the OSS auth middleware after `verify_*` populates `request.state`. `AuthContext` is a frozen Pydantic model with both fields required and discriminator-validated; the only constructor in `auth_service._build_auth_context_from_state` either builds the complete shape or returns `None`, in which case the middleware never publishes it on the ContextVar. Downstream handlers can therefore trust that `get_auth_context()` returns a fully-populated context or raises `AuthContextMissing` (caught upstream). Admin/Access requests deliberately don't produce an `AuthContext` (no tenant scope, no entitlement involvement). Several routers and workers (`permissions_router`, `commoners`, `organization_router`, `workspace_router`, `events/worker`) drop their `Request` plumbing and read the ambient context instead. `check_entitlements` and `scope_from` use the same source of truth. Four handlers that operate on a target org distinct from the caller's ambient scope project the entitlement check onto the target rather than letting it default: `ee.src.routers.organization_router.update_organization` passes `scope=scope_from(organization_id=UUID(organization_id))` (path-param org, `Flag.ACCESS`), `ee.src.utils.permissions.check_project_has_role_or_permission` passes `scope=scope_from(organization_id=project.organization_id)` (target project's org, `Flag.RBAC`), `oss.src.routers.organization_router.invite_user_to_organization` passes `scope=scope_from(organization_id=UUID(organization_id))` (`Gauge.USERS +1`), and `oss.src.routers.workspace_router.remove_user_from_workspace` passes `scope=scope_from(organization_id=project.organization_id)` (`Gauge.USERS -1`). Cross-org admin actions land the meter delta and the flag check in the right org instead of the caller's ambient plan.

**Shared DBAs.** The composed `ScopeDBA` in `oss.src.dbs.postgres.shared.dbas` (only consumer is `MeterDBA`) is relaxed to `nullable=True` for all four scope columns. The individual `OrganizationScopeDBA` / `WorkspaceScopeDBA` / `ProjectScopeDBA` / `UserScopeDBA` mixins keep `nullable=False` — other tenant-bound tables continue to require their scope.

## Migrations

Yes. **Schema migration** and **data migration** in a single transactional Alembic revision `9d3e8f0a1b2c_reshape_meters_table`:

- Schema: reshapes `meters` (new `meter_id` PK, new `workspace_id` / `project_id` / `user_id` / `day` columns, relaxes `organization_id` / `year` / `month` to nullable, rebuilds indexes).
- Data: deletes `APPLICATIONS` rows, remaps `TRACES` / `CREDITS` / `EVALUATIONS` rows to `TRACES_INGESTED` / `CREDITS_CONSUMED` / `EVALUATIONS_RUN`, backfills `meter_id` per row using the runtime canonicalizer, promotes `(year=0, month=0)` gauge sentinels to real `NULL`s.

The historical migration `7990f1e12f47_create_free_plans.py` was also patched (raw-SQL replay) so a fresh-DB upgrade still works without the now-removed `Gauge.APPLICATIONS` Python enum member.

## Tests and docs

Five unit-test files in `api/ee/tests/pytest/unit/` pin the new helpers: `test_compute_meter_id.py` (determinism, None-means-not-applicable, string/enum key equivalence, scope and period hierarchy validation, calendar rejection), `test_period_from.py` (`Period.None`/`YEARLY`/`MONTHLY`/`DAILY` shape, anchor honored on MONTHLY with year rollover, anchor ignored on DAILY), `test_scope_from.py` (unified-contract: `scope_from()` / `scope_from(scope=None)` / `scope_from(scope=Scope.ORGANIZATION)` are equivalent and project the ambient `AuthScope` at organization granularity; `scope=Scope.X` narrows to X; `organization_id=UUID(...)` is the no-ambient explicit branch; passing both raises; ambient projection populates exactly the dims each granularity implies), and `test_meters_dao_strict_soft.py` (pins `check`'s always-strict `current + delta <= limit` rule across delta sign, missing row, and `limit=None`; pins `adjust`'s strict-vs-non-strict predicate divergence by compiling the upsert statement and string-matching the WHERE clause — strict must emit `greatest(value + delta, 0) <= limit`, non-strict must emit `value < limit` without the delta term — plus the full predictable-overshoot truth table covering both modes: `delta > limit` short-circuits in Python without a DB call; `current=limit, delta>0` denies in both modes; `current<limit, current+delta>limit` denies in strict and allows in non-strict; `current+delta=limit` allows in both modes), and `test_meters_dao_fetch.py` (pins the per-dimension `IS NULL` binding when a structured `MeterScope`/`MeterPeriod` is supplied, the `MeterScope()` admin escape hatch on `organization_id`, the `scope=None` / `period=None` rollup escape hatches, and that MONTHLY reads bind `day IS NULL` so they cannot match DAILY rows). The manual exhaustive billing-period grid in `tests/manual/test_billing_period.py` was migrated from the deleted `compute_billing_period` to the new `monthly_period_from` (2-tuple return). `docs/designs/extend-meters/` carries the design story: `research.md` and `gap.md` as historical baselines, `proposal.md` and `tasks.md` describing what shipped, and `findings.md` with the deep-scan + PR-review results (55 review threads from eleven passes — 10× Copilot and 1× CodeRabbit — covering 45 distinct findings + 1 internal staging-deployment incident, all closed and resolved on GitHub or documented internally). The full test suite (`api/`, `services/`, `sdks/python/`) runs green: 1908 passing (1053 api + 143 services + 712 sdk), 9 unrelated skips, 23 xfails for live-LLM / live-webhook / deprecated-SDK fixtures.
