# Clean up meters — generalize scope/period, prepare global read limits

## What this unlocks

This PR turns the `meters` table from "one quota row per organization per month" into a general scope × period meter store. Any quota in the catalog can now declare a `scope` (organization / workspace / project / user) and a `period` (daily / monthly / yearly), and the meters layer stores one row per `(scope, period, key)` keyed by a deterministic `meter_id`.

The immediate capability this lights up is **per-user, per-day tracing read limits** (`Counter.TRACES_RETRIEVED`), declared structurally on every plan with `limit=None` for now — the wiring is in place, an operator can flip on a real limit per plan without any code change. Counter names also pick up an explicit verb (`TRACES_INGESTED`, `TRACES_RETRIEVED`, `CREDITS_CONSUMED`, `EVALUATIONS_RUN`) so the call site tells you which direction data is moving.

`/billing/usage` exposes `period` and `scope` additively. The frontend usage card renders the new fields. Stripe reporting is untouched — `TRACES_RETRIEVED` is not in the `REPORTS` allowlist, so it's tracked and surfaced internally and to customers but not billed.

The auth path now publishes a typed `AuthContext` on a `ContextVar` (organization / workspace / project / user IDs plus credentials), which removes the need to keep threading `Request` through services just to read those values, and lets entitlement checks default to the ambient caller's scope at whatever granularity the quota declares.

The catalog cleanup drops two dead `Counter` values (`EVALUATORS`, `ANNOTATIONS`), one dead `Gauge` (`APPLICATIONS`), and `Flag.HOOKS`. The previously never-incremented `EVALUATIONS` (only rolled back on failure, never charged) is renamed to `EVALUATIONS_RUN` and given proper `+1` / `-1` wiring at the three evaluation-run creation handlers (refund on internal failure). `TRACES_RETRIEVED` soft-check fires after every span/trace fetch or query (analytics, sessions, and users endpoints are deliberately excluded — they return aggregates or IDs, not trace data).

## What changed under the hood

**`meters` schema.** Columns added: `meter_id UUID` (new PK), `workspace_id`, `project_id`, `user_id` (UUID, nullable), `day SMALLINT` (nullable). `year` and `month` relaxed to nullable, the `(0, 0)` gauge sentinel promoted to real `NULL`s. The legacy composite PK is rebuilt as a non-unique secondary index extended with `day`, so the org-rollup scan path that `/billing/usage` and the entitlements soft-check use stays index-supported. `MeterDBE` keeps the FK to `subscriptions.organization_id` and nothing else — meters can legitimately outlive the workspaces/projects/users they were scoped to.

**`meter_id` canonicalization.** A single function `compute_meter_id(*, scope: MeterScope, period: MeterPeriod, key)` in `ee.src.core.meters.types` is the only thing that produces a `meter_id`. The DAO, helpers, and the Alembic backfill all go through it; the database has no SQL-side mirror. The canonical string excludes `None` fields, uses short alphabetically-sorted labels (`org`, `wrk`, `prj`, `usr`, `y`, `m`, `d`, `key`), and is hashed against a namespace UUID derived from a literal `"agenta"` string (`uuid5(NAMESPACE_DNS, "agenta")` lives in `env.agenta.uuid_namespace`) so the IDs survive auth-key rotations. `MeterScope` validates the scope hierarchy, `MeterPeriod` validates the calendar (Feb 30 is rejected), and `MeterDTO` constructs both internally so any invalid shape fails at DTO construction — there is no way to bypass canonicalization.

**Migration.** One transactional Alembic revision (`9d3e8f0a1b2c_reshape_meters_table`) does the enum reshape and the schema reshape together. The Postgres `meters_type` enum is type-swapped (not `ALTER TYPE ADD VALUE`) so renames, additions, and removals happen atomically: `APPLICATIONS` rows are deleted, `TRACES`/`CREDITS`/`EVALUATIONS` rows are mapped to their verb-explicit successors via a `CASE` in the `USING` clause, `TRACES_RETRIEVED` is added. The legacy composite PK is dropped before `year`/`month` are relaxed (Postgres refuses `DROP NOT NULL` on a PK column); `meter_id` is then backfilled by importing the same canonicalizer the runtime uses. Downgrade is symmetric: drop new PK and index, drop new columns, restore `year`/`month` to `NOT NULL DEFAULT 0`, reverse the enum swap, then recreate the legacy composite PK last. The historical `7990f1e12f47_create_free_plans` migration is updated to use raw SQL with a `_LEGACY_APPLICATIONS_KEY = "APPLICATIONS"` constant so it no longer depends on the (now-removed) `Gauge.APPLICATIONS` Python enum member at import time.

**`Quota` and `Probe` reshape.** `Quota.monthly: bool` is replaced by `Quota.period: Optional[Period]` (where `Period` is the new `DAILY` / `MONTHLY` / `YEARLY` enum); `Quota.scope: Optional[Scope]` is added with the convention that `None` means `ORGANIZATION`. `Quota.strict` and `Probe.delta` move from `Optional[bool] = False` to `Optional[bool] = None` (None means False), and the previous minute-valued `Periods(str, Enum)` is renamed to `Retention(int, Enum)` so `Quota.retention: Optional[Retention]` flows straight through. `Probe.monthly` becomes `Probe.period`. Every existing plan declaration is migrated mechanically.

**DAO layer.** `MetersDAO.dump` / `bump` / `fetch` / `check` / `adjust` all key on `meter_id`. `adjust` uses `ON CONFLICT (meter_id) DO UPDATE` — single-column conflict target instead of the old four-column tuple. `fetch` takes `MeterScope` and `MeterPeriod` directly and applies `filter_by` per non-`None` dimension, so the same method serves org-rollup scans (`scope_from(organization_id=...)`) and pinpoint reads (`scope_from(scope=Scope.USER)`) without branching. Period normalization happens via a `with_period` helper on `MeterDTO` that returns a new instance with a recomputed `meter_id` rather than mutating in place. The old `compute_billing_period` was replaced by two helpers in `ee.src.utils.entitlements`: `monthly_period_from` (honors Stripe-style anchor day) and `period_from` (builds a `MeterPeriod` for the current moment at the requested granularity).

**Entitlements helper.** `check_entitlements(*, key, delta=None, cache=False, scope=None, period=None)` is the public surface. When `scope`/`period` are omitted, the helper projects the ambient `AuthScope` (from the new `ContextVar`) down to the granularity declared by `quota.scope`, and computes the current bucket via `period_from(period=quota.period, anchor=...)`. Cache keys are scope+period+key projections rather than the previous `(organization_id, key, year, month)` tuple. The error policy is now explicit: `EntitlementsException` (config bugs — invalid key, missing plan, no subscription) propagates; everything else fails open, so a meter glitch never blocks a request. `MetersService` and `SubscriptionsService` are registered at the EE composition root via `register_entitlements_services(...)` instead of being imported into the entitlements module (which would create a circular dependency through the meters DAO).

**Write-side enforcement (`EVALUATIONS_RUN`).** Three handlers — `EvaluationsRouter.create_runs`, `SimpleEvaluationsRouter.create_evaluation`, `SimpleQueuesRouter.create_simple_queue` — check the quota after the permission check, return `429 Too Many Requests` on `not allowed`, and refund with `delta=-N` if the service call raises. The refund triggers on any `Exception` (broad-safety trade-off — domain exceptions also refund, which is the conservative choice). No domain exception or HTTP exception subclass was introduced; the router translates the boolean directly.

**Read-side enforcement (`TRACES_RETRIEVED`).** The six read paths — `query_spans`, `fetch_spans`, `fetch_span`, `query_traces`, `fetch_traces`, `fetch_trace` — call `check_entitlements(key=Counter.TRACES_RETRIEVED, cache=True, delta=...)` after the result is known. Delta is `len(traces)` for trace-shaped responses, `len({s.trace_id for s in spans})` for span-shaped responses, and `1` / `0` for singletons. With `limit=None` everywhere today the check is a no-op; setting a real limit on any plan turns it on without further code change. Analytics, sessions, and users endpoints are explicitly excluded — they return aggregates or IDs, not trace data.

**Auth + context wiring.** A new `AuthContext` Pydantic model (credentials + `AuthScope`) is published on a `ContextVar` by the OSS auth middleware after `verify_*` populates `request.state`. Admin/Access requests deliberately don't produce an `AuthContext` (no tenant scope, no entitlement involvement). Several routers and workers (`permissions_router`, `commoners`, `organization_router`, `workspace_router`, `events/worker`) drop their `Request` plumbing and read the ambient context instead. `check_entitlements` and `scope_from` use the same source of truth.

**Shared DBAs.** The composed `ScopeDBA` in `oss.src.dbs.postgres.shared.dbas` (only consumer is `MeterDBA`) is relaxed to `nullable=True` for all four scope columns. The individual `OrganizationScopeDBA` / `WorkspaceScopeDBA` / `ProjectScopeDBA` / `UserScopeDBA` mixins keep `nullable=False` — other tenant-bound tables continue to require their scope.

**Tests and docs.** `api/ee/tests/pytest/unit/test_compute_meter_id.py` covers determinism, the None-means-not-applicable invariant, string/enum key equivalence, scope and period hierarchy validation, and calendar rejection. `docs/designs/extend-meters/` carries the design story: `research.md` and `gap.md` as historical baselines, `proposal.md` and `tasks.md` describing what shipped, and `findings.md` with the deep-scan results (all closed).
