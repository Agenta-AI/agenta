# Writing instructions and skills using the Diátaxis framework

The **Diátaxis** framework describes four complementary forms of documentation—tutorials, how‑to guides, technical reference and explanation—and emphasises that each serves a distinct purpose [oai_citation:0‡diataxis.fr](https://diataxis.fr/tutorials-how-to/#:~:text=The%20difference%20between%20a%20tutorial,to%20guide%C2%B6).  When you write instructions or “skills” for an agent, you are essentially writing **how‑to guides**: concise, goal‑oriented directions that help the user accomplish a specific task [oai_citation:1‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=How).  This document summarises the Diátaxis principles and offers guidance on writing effective instructions for code‑review agents.

## The four documentation types

Diátaxis structures documentation along two dimensions: *study vs work* and *skill vs knowledge*.  Placing documents in the wrong quadrant leads to confusion and poor user experience [oai_citation:2‡diataxis.fr](https://diataxis.fr/map/#:~:text=As%20a%20map%2C%20it%20places,between%20them%20highlight%20their%20distinctions).  Below is a high‑level overview of each type:

### Tutorials – learning through practice

- **Purpose:** Offer a *learning experience* for users who are acquiring a new skill [oai_citation:3‡diataxis.fr](https://diataxis.fr/tutorials-how-to/#:~:text=Di%C3%A1taxis%20insists%20that%20what%20matters,to%20guides).  They walk the learner through tasks step by step and provide safety; the tutor takes responsibility if the learner gets into trouble [oai_citation:4‡diataxis.fr](https://diataxis.fr/tutorials-how-to/#:~:text=world%2C%20and%20anything%20could%20appear,to%20disrupt%20the%20journey).  
- **Characteristics:** Occur in a contrived environment where tools and conditions are set up to guarantee success [oai_citation:5‡diataxis.fr](https://diataxis.fr/tutorials-how-to/#:~:text=The%20tutorial%20takes%20place%20in,to%20ensure%20a%20successful%20experience).  The path is linear and does not branch [oai_citation:6‡diataxis.fr](https://diataxis.fr/tutorials-how-to/#:~:text=A%20tutorial%E2%80%99s%20path%20follows%20a,doesn%E2%80%99t%20offer%20choices%20or%20alternatives).  
- **When to use:** When the audience is learning and needs guidance to acquire competence.  Tutorials should *not* be embedded in how‑to guides, which assume competence and focus on work [oai_citation:7‡diataxis.fr](https://diataxis.fr/tutorials-how-to/#:~:text=The%20user%20following%20a%20how,to%20complete%20a%20particular%20task).

### How‑to guides – accomplishing tasks (instructions)

- **Purpose:** Guide an already‑competent user through a real‑world task or problem [oai_citation:8‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=How,oriented).  They show the user how to get from their current situation to the desired result [oai_citation:9‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=How,oriented).  
- **Characteristics:** Focus on the user’s goal, not on the machinery.  They contain only actions—no digressions, explanation or teaching [oai_citation:10‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Key%20principles%C2%B6).  They may branch to handle multiple conditions (“If this, then that…”) and must prepare for unexpected situations [oai_citation:11‡diataxis.fr](https://diataxis.fr/tutorials-how-to/#:~:text=The%20how,how%20to%20deal%20with%20it).  
- **Audience:** A user *at work* who already knows what they want to achieve and has the necessary basic skills [oai_citation:12‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Key%20principles%C2%B6).  
- **Examples:** “How to configure reconnection back‑off policies,” “How to integrate application performance monitoring.”

### Technical reference – factual information

- **Purpose:** Provide complete, accurate and neutral facts a user needs while applying their skills [oai_citation:13‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=The%20real%20test%2C%20though%2C%20if,want%20to%20think%20about%20it).  Reference is consulted *during work* to look up details but does not instruct [oai_citation:14‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=Work%20and%20study%C2%B6).  
- **Characteristics:** Boring, unmemorable and dry; often lists, tables, API descriptions or configuration parameters [oai_citation:15‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=There%20are%20good%20rules%20of,thumb).  They should not contain narrative or explanation; any explanatory digressions dilute reference [oai_citation:16‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=It%20usually%20happens%20while%20writing,how%20it%20came%20to%20be).  
- **When to use:** When users need factual information (e.g., full list of options or parameters) rather than instructions or background.  A how‑to guide may link to reference material but should not include reference content inline [oai_citation:17‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Key%20principles%C2%B6).

### Explanation – context and reasoning

- **Purpose:** Answer “why?” by providing background, rationale and conceptual understanding [oai_citation:18‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=Explanation%20and%20reference%20both%20belong,reader%2C%20they%20contain%20theoretical%20knowledge).  Explanation helps readers acquire knowledge and understanding rather than act [oai_citation:19‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=Work%20and%20study%C2%B6).  
- **Characteristics:** Discursive, sometimes narrative; often suitable for reading in non‑work settings [oai_citation:20‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=,might%20read%20in%20the%20bath).  Unlike reference, explanation need not be exhaustive; its goal is to illuminate concepts.  
- **When to use:** To give theoretical context or design rationale.  How‑to guides may link to explanatory articles when readers need to understand background concepts, but explanations should not appear inside instructions [oai_citation:21‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=As%20a%20result%20one%20often,allowed%20to%20develop%20appropriately%20and).

## Why distinctions matter

Conflating these types is one of the most common problems in documentation.  Tutorials and how‑to guides both present ordered steps [oai_citation:22‡diataxis.fr](https://diataxis.fr/tutorials-how-to/#:~:text=In%20important%20respects%2C%20tutorials%20and,is%20to%20know%20or%20understand), but they serve different needs: a tutorial teaches skills while a how‑to guide helps an expert apply skills to a task [oai_citation:23‡diataxis.fr](https://diataxis.fr/tutorials-how-to/#:~:text=Di%C3%A1taxis%20insists%20that%20what%20matters,to%20guides).  Similarly, reference and explanation both deliver factual or theoretical knowledge [oai_citation:24‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=Explanation%20and%20reference%20both%20belong,reader%2C%20they%20contain%20theoretical%20knowledge), yet reference is used during work, whereas explanation is for study [oai_citation:25‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=Work%20and%20study%C2%B6).  Mixing modes leads to digressions that hinder the reader: reference polluted with explanation becomes harder to scan, and explanations buried inside how‑to guides break the flow [oai_citation:26‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=It%20usually%20happens%20while%20writing,how%20it%20came%20to%20be).  Always ask: **Is the user trying to learn or to work? Do they need procedures, facts or context?**

## Writing effective how‑to guides (instructions)

When writing instructions or “skills,” you are creating how‑to guides.  The Diátaxis framework provides detailed guidance on how to craft them:

### Focus on the user’s goal

A how‑to guide must be written from the user’s perspective and address a real problem or goal [oai_citation:27‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=How).  Avoid defining guides by the operations of the tool; instead, define them by what the user needs to accomplish [oai_citation:28‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=,%E2%80%9D).  Always frame the title and scope around the user’s project, not the product feature.  Good titles explicitly state the goal (e.g., “How to integrate application performance monitoring”) [oai_citation:29‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Pay%20attention%20to%20naming%C2%B6).

### Assume competence and concentrate on actions

How‑to guides serve already‑competent users.  They assume the reader knows what they want and can follow instructions [oai_citation:30‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Key%20principles%C2%B6).  Your guide should contain *only* actions necessary to achieve the goal [oai_citation:31‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Key%20principles%C2%B6).  Resist the temptation to teach or to include every possible option—provide links to tutorials or reference material instead [oai_citation:32‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Key%20principles%C2%B6).  The language should use conditional imperatives: “If you want *x*, do *y*” [oai_citation:33‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=This%20guide%20shows%20you%20how,to%E2%80%A6).

### Address real‑world complexity and remain adaptable

Real‑world problems are rarely linear.  How‑to guides must prepare for unexpected conditions by describing branches or alternatives (e.g., “If this, then that”) [oai_citation:34‡diataxis.fr](https://diataxis.fr/tutorials-how-to/#:~:text=The%20how,how%20to%20deal%20with%20it).  They should also remain open to a range of possibilities; avoid overly narrow guides that apply only to one exact scenario [oai_citation:35‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Address%20real).  Acknowledge where judgement is needed—solving a problem involves thinking as well as doing [oai_citation:36‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=%E2%80%9CActions%E2%80%9D%20in%20this%20context%20includes,as%20what%20the%20user%20does).

### Omit the unnecessary

Practical usability is more valuable than completeness in instructions [oai_citation:37‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Omit%20the%20unnecessary%C2%B6).  Unlike tutorials, how‑to guides do not need to cover every step from beginning to end; they can start at a meaningful point and assume the user can join it to their own work [oai_citation:38‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Omit%20the%20unnecessary%C2%B6).  Cut extraneous steps, explanations or reference material that might interrupt flow [oai_citation:39‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Key%20principles%C2%B6).

### Provide a logical sequence and seek flow

A how‑to guide is a sequence of actions that follow a logical order [oai_citation:40‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Provide%20a%20set%20of%20instructions%C2%B6).  Sometimes this order is dictated by dependencies (step 2 requires step 1).  Sometimes it is about pacing the user’s thinking so that each action prepares the context for the next [oai_citation:41‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Sometimes%20the%20need%20is%20more,reason%20for%20putting%20it%20first).  Pay attention to flow—the smooth progression of thought and action [oai_citation:42‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Seek%20flow%C2%B6).  Avoid forcing the user to jump back and forth between contexts or hold too many concepts in mind [oai_citation:43‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Again%2C%20this%20can%20be%20somewhat,is%20this%20necessary%20or%20avoidable).  The best guides anticipate the user’s needs, like a helper who has the next tool ready [oai_citation:44‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=At%20its%20best%2C%20how,place%20it%20in%20your%20hand).

### Use clear, conditional language

State the problem or goal at the outset [oai_citation:45‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=The%20language%20of%20how).  Use conditional imperatives: “If you want *x*, do *y*” [oai_citation:46‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=This%20guide%20shows%20you%20how,to%E2%80%A6).  Avoid imperative commands without context (e.g., “press Deploy” by itself) [oai_citation:47‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=,%E2%80%9D).  Encourage linking to reference for full option lists [oai_citation:48‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=This%20guide%20shows%20you%20how,to%E2%80%A6).

### Learn from recipes

A recipe is an excellent model: it clearly defines what will be achieved and lists ingredients and steps [oai_citation:49‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Applied%20to%20food%20and%20cooking%C2%B6).  Recipes assume basic competence; they are not cooking lessons.  Similarly, how‑to guides should not attempt to teach novices, and they should avoid distracting digressions about context [oai_citation:50‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Applied%20to%20food%20and%20cooking%C2%B6).

## Writing other documentation types

While instructions are how‑to guides, you may also need to create tutorials, reference or explanations.  Keep them separate and follow these guidelines:

- **Tutorials:** Provide a safe, structured learning experience.  Eliminate unexpected events; assume responsibility for the learner; show exactly what to do and why [oai_citation:51‡diataxis.fr](https://diataxis.fr/tutorials-how-to/#:~:text=world%2C%20and%20anything%20could%20appear,to%20disrupt%20the%20journey).  Use a linear sequence and avoid branches.  Suitable for training new reviewers on fundamental skills.

- **Reference:** Deliver complete and accurate facts that users need during work [oai_citation:52‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=The%20real%20test%2C%20though%2C%20if,want%20to%20think%20about%20it).  Use tables, lists and neutral descriptions.  Do not embed procedures or narrative [oai_citation:53‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=It%20usually%20happens%20while%20writing,how%20it%20came%20to%20be).  Suitable for listing review severity levels, file formats or configuration options.

- **Explanation:** Provide background, concepts or rationale [oai_citation:54‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=Explanation%20and%20reference%20both%20belong,reader%2C%20they%20contain%20theoretical%20knowledge).  It belongs to the “study” side: readers use it to understand why things are the way they are [oai_citation:55‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=Work%20and%20study%C2%B6).  Use narrative style and examples.  Suitable for justifying review policies or explaining the philosophy behind vulnerability classification.

When deciding where content belongs, ask whether it is used by someone learning or someone doing work [oai_citation:56‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=The%20real%20test%2C%20though%2C%20if,want%20to%20think%20about%20it).  Keep each mode free of the content of the others [oai_citation:57‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=It%20usually%20happens%20while%20writing,how%20it%20came%20to%20be).

## Quality: functional and deep

High‑quality instructions must achieve both **functional** and **deep** quality.  Functional quality means documentation is accurate, complete, consistent, useful and precise [oai_citation:58‡diataxis.fr](https://diataxis.fr/quality/#:~:text=Functional%20quality%C2%B6).  Each dimension is independent—documentation can be accurate yet incomplete or vice versa [oai_citation:59‡diataxis.fr](https://diataxis.fr/quality/#:~:text=These%20properties%20of%20functional%20quality,complete%2C%20consistent%20and%20also%20useless).  Deep quality concerns intangible attributes: instructions should feel good to use, have flow, anticipate the user’s needs and be beautiful [oai_citation:60‡diataxis.fr](https://diataxis.fr/quality/#:~:text=Think%20of%20characteristics%20such%20as%3A).  Deep quality cannot be measured directly; it is judged by how the user experiences the document [oai_citation:61‡diataxis.fr](https://diataxis.fr/quality/#:~:text=).  Deep quality depends on functional quality; no one will experience a guide as delightful if it is inaccurate or inconsistent [oai_citation:62‡diataxis.fr](https://diataxis.fr/quality/#:~:text=And%2C%20deep%20quality%20is%20conditional,experience%20of%20documentation%20is%20tarnished).  Strive for both: maintain discipline and precision while crafting guides that flow smoothly and fit human thinking [oai_citation:63‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Seek%20flow%C2%B6).

## Process: using Diátaxis to improve documentation

Diátaxis is not merely a classification system; it is a guide for iterating on documentation [oai_citation:64‡diataxis.fr](https://diataxis.fr/how-to-use-diataxis/#:~:text=Di%C3%A1taxis%20as%20a%20guide%20to,work%C2%B6).  When improving instructions:

1. **Use Diátaxis as a guide, not a rigid plan.**  The framework provides a map to help you understand what each piece of documentation is doing, not a strict checklist [oai_citation:65‡diataxis.fr](https://diataxis.fr/how-to-use-diataxis/#:~:text=Use%20Di%C3%A1taxis%20as%20a%20guide%2C,not%20a%20plan%C2%B6).  Don’t create empty sections for each type; let the structure emerge as you improve existing content [oai_citation:66‡diataxis.fr](https://diataxis.fr/how-to-use-diataxis/#:~:text=Don%E2%80%99t%20worry%20about%20structure%C2%B6).
2. **Iterate in small steps.**  Avoid grand restructures.  Assess a small piece of documentation, identify the user need it serves and decide a single next action to improve it [oai_citation:67‡diataxis.fr](https://diataxis.fr/how-to-use-diataxis/#:~:text=Work%20one%20step%20at%20a,time%C2%B6).  Publish or commit each small improvement rather than waiting for a perfect overhaul [oai_citation:68‡diataxis.fr](https://diataxis.fr/how-to-use-diataxis/#:~:text=Work%20one%20step%20at%20a,time%C2%B6).
3. **Challenge each piece:** Ask: *What user need is represented?*  *How well does it serve that need?*  *Does the language meet the requirements of the chosen documentation mode?* [oai_citation:69‡diataxis.fr](https://diataxis.fr/how-to-use-diataxis/#:~:text=Assess%20it,Do%20its%20language%20and%20logic).  Move, remove or add material to match the correct type.
4. **Let structure develop organically.**  As you refine individual “cells,” the overall structure will naturally take on the Diátaxis form [oai_citation:70‡diataxis.fr](https://diataxis.fr/how-to-use-diataxis/#:~:text=Allow%20your%20work%20to%20develop,organically%C2%B6).

## Applying Diátaxis to code‑review skills and instructions

When you design internal skills for a code‑review agent (e.g., `CR.instructions.md`), treat them as how‑to guides:

1. **Define the user’s goal clearly.**  Each instruction file should correspond to a specific task (e.g., *“evaluate pull request for security vulnerabilities”*).  Avoid vague titles like “Code review” and instead use titles that state the desired outcome [oai_citation:71‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Pay%20attention%20to%20naming%C2%B6).
2. **Assume competence.**  The agent is your already‑trained reviewer.  Instructions should not teach basic concepts or justify why something matters.  Reserve training material for separate tutorials or explanatory notes.  Link to these resources if the agent needs background [oai_citation:72‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Key%20principles%C2%B6).
3. **Provide a sequence of actions.**  Describe the steps the agent should take: set up the environment, analyse scope, evaluate code patterns, record findings, suggest remediations, and generate a report.  Use conditional branches to handle different scenarios (e.g., if tests fail, perform additional analysis) [oai_citation:73‡diataxis.fr](https://diataxis.fr/tutorials-how-to/#:~:text=The%20how,how%20to%20deal%20with%20it).
4. **Integrate remediation within each finding.**  When the agent records a vulnerability or risk, it should also record the explanation, evidence references and suggested remediations in the same entry, rather than delegating fixes to a separate file.  This aligns with the idea that instructions should help the user (agent) complete their work in one place.
5. **Keep reference separate.**  Place lists of severity levels, rule IDs, or configuration options in separate reference files (e.g., `CR.deliverables.md` or a rubric file).  Link to them from instructions rather than embedding them [oai_citation:74‡diataxis.fr](https://diataxis.fr/reference-explanation/#:~:text=It%20usually%20happens%20while%20writing,how%20it%20came%20to%20be).
6. **Use conditional language and anticipate decisions.**  Frame instructions as “If you find an injection vulnerability, include remediation option A or B” rather than unconditional commands.  Provide multiple remediation options where appropriate, because there may be several valid ways to fix an issue.
7. **Prioritise flow and reduce cognitive load.**  Organise instructions so that each step naturally leads to the next.  Avoid unnecessary context switching; group related tasks together [oai_citation:75‡diataxis.fr](https://diataxis.fr/how-to-guides/#:~:text=Again%2C%20this%20can%20be%20somewhat,is%20this%20necessary%20or%20avoidable).  Where the agent must switch context (e.g., from scanning code to writing a report), signal this clearly.
8. **Iterate and refine.**  Start with a minimal set of instructions.  After each review run, evaluate where the agent hesitated or misinterpreted instructions and refine them.  This iterative approach mirrors the organic growth recommended by Diátaxis [oai_citation:76‡diataxis.fr](https://diataxis.fr/how-to-use-diataxis/#:~:text=Work%20one%20step%20at%20a,time%C2%B6).

## Conclusion

Diátaxis offers a clear framework for structuring and writing documentation.  By distinguishing between tutorials, how‑to guides, reference and explanations, you ensure that each document meets a specific user need.  For code‑review skills, focus on creating concise, goal‑oriented how‑to guides: assume competence, provide only actions, handle real‑world complexity, and maintain flow.  Support them with separate reference and explanation materials, and refine them iteratively.  Combining functional quality (accuracy, completeness, consistency) with deep quality (flow, anticipation, beauty) will produce instructions that not only work but feel good to use [oai_citation:77‡diataxis.fr](https://diataxis.fr/quality/#:~:text=Functional%20quality%C2%B6).  Adopting Diátaxis in this way will help your documentation evolve organically, serving both the agent and stakeholders effectively.
